the concepts
Concept 01 // The definition

🌐 What even IS a web application?

A web application is a program — just like Excel or Photoshop — except you don't install it. You access it through a web browser.

📝 Think of it like Netflix vs. a Blu-ray disc

A traditional program is a Blu-ray — you buy it, insert it, install it, run it locally. Each copy lives on one computer.

A web application is Netflix — the actual movie lives on a server somewhere. Your "screen" (browser) just streams it. Every device with a browser gets the latest version automatically.

The browser is the universal runtime. Chrome, Firefox, Safari — pick any, the app just works.

Examples of web applications you use daily

  • 📧 Webmail — Gmail, Outlook, Proton Mail
  • 📊 Office suites — Google Docs, Microsoft 365, Zoho
  • 🛒 Online shopping — Amazon, AliExpress, Etsy
  • 🏦 Online banking — your bank's website
  • 🎬 Streaming — Netflix, YouTube, Spotify web player
  • 💬 Social media — Twitter/X, Reddit, Instagram web

the data
type & click → ← search query → ← rows
💻 CLIENT (your browser)

What it does: Runs HTML, CSS, and JavaScript. Collects your input (forms, clicks, keystrokes) and sends it to the server.

Attack surface: XSS (JavaScript running in YOUR context), CSRF (tricking your browser into making requests), clickjacking.

Tools attackers use: Browser DevTools (F12), Burp Suite, curl.

The 4-step search flow

1. User types "red tires" → browser sends it to the web app. 2. Web app queries the products database for matching items. 3. Database returns matching rows to the app. 4. App formats the results as HTML and sends it back to the browser.
🎯 Security insight: Every one of those 4 steps is an attack surface. Step 1 can be abused with XSS. Step 2 with SQL injection. Step 3 with data leakage. Step 4 with XSS-in-output. Four opportunities to break something per single user search.
🧠 Brain Power
If the client is YOUR browser, can you trust anything that comes from it? Why or why not?

Never trust the client. Ever. It's on the user's machine — they (or an attacker) can change anything before it reaches the server. JavaScript validation? Disabled in DevTools. Hidden form fields? Edited in Burp. "Read-only" inputs? LOL.

This is the single most important lesson in web security: validate EVERYTHING on the server. Again. Even if the client said it was fine.

Concept 04 // The categories

📚 The 4 OWASP classes THM introduces here

OWASP publishes a "Top 10" list of the biggest web vulnerability categories. This room covers 4 of them. Know these four by heart.

A01

🔓 Broken Access Control

Users can access stuff they shouldn't. The #1 web vuln in the world.

Example: Change /user/16 to /user/17 in the URL and you're reading someone else's profile. That's IDOR — you'll exploit one in Task 3 of this room.
A02

🔐 Cryptographic Failures

Sending / storing sensitive data without proper encryption.

Example: A login page served over plain http:// — anyone sniffing the Wi-Fi sees the password in cleartext. Use HTTPS always.
A03

💉 Injection

Attacker sneaks code into user input; the server executes it.

Example: Search field accepts ' OR 1=1-- → SQL injection dumps the whole users table. Classic, evergreen.
A07

🔑 Authentication Failures

Login is broken or easy to bypass.

Example: No account lockout + no CAPTCHA + weak password policy = brute-force heaven. Tools like Hydra test 1000 passwords per minute.
💡 These are categories, not bugs. Within each category live dozens of specific bug types. "Access Control" includes IDOR, directory traversal, missing /admin auth check, horizontal privesc, vertical privesc, etc. Learn the category first; the specific bugs become variations on a theme.
Concept 05 // The practical attack

🎣 IDOR — the change-a-number attack

Task 3 drops you into a live web app. You exploit an IDOR to find which user sabotaged the inventory. Here's how IDOR actually works.

📝 Think of it like hotel rooms with the same key pattern

You check into Room 216. Your key card also happens to work on Rooms 217, 218, 219 — because the hotel programmed the cards sequentially and forgot to check "does this guest belong to THIS room?"

IDOR is when a website uses an ID to look up data but forgets to check: "wait, is this person ALLOWED to see this data?"

The URL tell

Look for URLs (or request parameters) with sequential-looking IDs:

# Your profile https://store.tryhackme.thm/customers/user?id=16 # 🤔 what if I change 16 to 17? https://store.tryhackme.thm/customers/user?id=17 # another user's profile # Product 52 — but what about 51, 53? https://store.tryhackme.thm/products/product?id=52

🎮 Try it — IDOR Simulator

Change the id in the URL. Watch the profile change. See what a real IDOR feels like.

https://store.tryhackme.thm/customers/user?id=
🧠 Brain Power
If IDOR is "just changing a number in the URL," why is it one of the highest-paying classes of bugs in bug bounty?

Because it's everywhere AND devs miss it constantly:

  • Automated scanners can't find it easily — they don't know what "should" be accessible to whom.
  • Every feature that shows user-specific data is a potential IDOR (profiles, messages, orders, documents, medical records).
  • When it hits PII or financial data, the impact is massive — GDPR/HIPAA fines, lawsuits, breach notifications.

HackerOne has paid out 5-figure bounties for single IDOR bugs at Uber, Shopify, Airbnb. It's the bug that pays the bills.

⚠️ Legal reminder: IDOR testing is ONLY legal when you have bug bounty scope or written permission. Changing IDs on a random website to "just see what happens" = unauthorized access. The tech is easy; the ethics/legality is the part that matters.

TASK 3

Practical Example: IDOR Lab

Click "View Site" → Inventory Management System. Someone sabotaged the tire assignments. Find the saboteur via IDOR, revert the changes, get the flag.

▸ Show walkthrough + reasoning
🎯 The steps:
  1. Click "View Site" to open the Inventory Management System.
  2. Look at the URL — you're viewing one user's activity. Notice the id= (or similar) parameter.
  3. Try changing the ID to other numbers: 1, 2, 3... — each one shows a different user's activity.
  4. Find the user whose activity shows them mis-assigning tires (scooter tires to bikes, etc.) — that's the saboteur.
  5. Revert their changes using the UI. A flag appears.
Q: What is the flag?
[[MACHINE — format: THM{____________}]]
Fill in when you run the lab. Check the page after reverting the sabotage.
WHY: The website trusts whatever id you send. It never checks "is this user the same as the logged-in user?" — so you can browse through every user's activity. This is IDOR in its purest form. The fix would be to check the session cookie matches the requested user ID on every request.

🎯 Bug bounty scenario

You're logged into a site as user 42. The URL of your profile is /api/users/42. You change it to /api/users/43 and it returns user 43's email and phone number. What did you just find?
SQL Injection — you tricked the database.
XSS — you injected JavaScript.
IDOR — a form of Broken Access Control. The server doesn't check if YOU are user 43.
Cryptographic Failure — the data wasn't encrypted.

🎯 Mental model

Where, in the 3-tier architecture, should data validation ALWAYS happen?
Only in the client (browser JavaScript).
Only in the database.
On the server — always. Client validation is UX, not security.
It doesn't matter as long as HTTPS is enabled.
