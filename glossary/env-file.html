📄 .env File

Where devs keep their secrets during development.
Also where devs accidentally upload them to the world.

"One .env leak = 5–15 stolen credentials in a single file."
Chapter 01 // Definition

📖 What is a .env file?

A plain-text file, usually named literally .env, sitting in the root of a project. Each line is KEY=VALUE. That's it. No encryption, no structure, no magic.

📝 Think of it as a sticky note

Imagine your dev setup is a laptop with sticky notes around the monitor. Each note says something like:

📝 Database password: hunter2
📝 Stripe key: sk_live_xyz
📝 AWS key: AKIA...

They're super convenient — the laptop apps read them at startup and "just work." But if someone photographs your monitor, every secret is exposed at once.

A .env file is that monitor wall. git push is the photograph.

In code, libraries like python-dotenv, dotenv (Node), or Dotenv (Ruby) load the file at startup and drop each pair into the process's environment variables. Your app reads them with os.getenv("STRIPE_KEY") — clean, no hardcoding.

why it matters
Chapter 02 // The leak math

💥 Why .env files are the most valuable leak

💰 Density

A single .env typically has 5–15 credentials. Finding ONE file = finding many keys at once.

🎯 Related creds

All credentials belong to the SAME app. DB creds + API keys + SMTP in one bundle = full takeover chain.

🤷 Low detection

Individual keys may dodge bug bounty scanners. An .env announces itself by filename.

📂 Easy to hunt

GitHub query: filename:.env DB_PASSWORD. Millions of results. Humans can't triage them all — that's the gap.

real world
Chapter 03 // Seen in the wild

📰 Actual .env-caused breaches

🏪 Shopify merchants (ongoing)

Thousands of third-party Shopify app developers commit .env files with SHOPIFY_API_SECRET, giving attackers full access to every merchant that installed the app.

🎮 Grand Theft Auto VI (2022)

The Rockstar leak started with an attacker finding developer credentials in a Slack message, then pivoting through a dev .env to pull 90 videos + source code.

💳 Toyota (2023)

A public GitHub repo exposed .env-style configs including a Google Cloud access key for 10 years. 260,000 customer records leaked.

🪙 Kyber Network (2023)

A DeFi exchange lost $265K after an attacker found Google Tag Manager credentials in a leaked config, then injected malicious JS into the site.

Chapter 04 // How to not leak one

🛡️ The prevention cheat sheet

  • Add to .gitignore FIRST (before you create the .env):
    .env .env.* !.env.example
  • Use .env.example with placeholder values, commit THAT. New devs copy it to .env and fill in real values locally.
  • Install gitleaks as a pre-commit hook — it blocks commits containing secret-shaped strings.
  • Use a secret manager in production — AWS Secrets Manager, HashiCorp Vault, Doppler. .env is for local dev only.
  • Audit existing repos: git log --all --full-history -- .env to see if .env was ever committed in history.
  • If you leaked one: rotate every single credential in it, rewrite git history with git filter-repo, force push, contact GitHub Support to purge caches.

🎯 Quick Check

A developer commits a .env to a public repo, notices 10 minutes later, and deletes it in a new commit. Is the secret safe now?
Yes — the file is gone from the current version.
Yes, as long as they make the repo private afterwards.
No — git history keeps the old commit. Anyone who cloned or a scanner that indexed the repo already has the secret. Every credential in the .env must be rotated.
No — but only if the repo had more than 100 stars.
