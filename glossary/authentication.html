Authentication 🔑

The process of proving you are who you say you are.
The login form. The SSH key. The TOTP code.

"I am Talha, and I can prove it." — every login, ever. ☕
Chapter 01 // Definition

📖 What is authentication?

Authentication (authn) is the process of verifying identity — confirming someone is who they claim to be before granting access.

📝 Think of it like showing ID at the airport

The TSA agent doesn't know you. You hand them your passport. They verify the photo matches your face. That's authentication — you proved your claimed identity.

What the agent does NEXT (can you board this flight? are you on the no-fly list?) is authorization — a separate check we call access control.

The three authentication factors

🧠 Something you KNOW

Password, PIN, security question. Memorizable. Also guessable.

📱 Something you HAVE

Phone (for SMS/TOTP codes), hardware key (YubiKey), smart card. Physical possession.

🫆 Something you ARE

Fingerprint, face, voice, retina. Biometrics. Hard to change when stolen.

💡 Multi-factor authentication (MFA / 2FA): Requires TWO different factors. Password (know) + phone code (have) = 2FA. Much harder to break because attackers need both.
Chapter 02 // Where it breaks

💥 OWASP A07 — Authentication Failures

When authentication breaks, attackers get in as someone they're not. Here's how it breaks:

🔨 No rate limiting

Login accepts unlimited attempts. Brute-force runs wild.

📜 Credential stuffing

Attackers take username+password pairs leaked from OTHER breaches and try them here. Password reuse = you're already pwned.

🙈 Weak password policy

password, 123456, qwerty still top the most-used lists every year. No enforcement = easy wins.

💾 Plaintext password storage

Database gets breached → every password leaks in clear. No bcrypt, no argon2, just raw strings. 2010s-era mistakes that STILL happen.

🪄 Session hijacking

Cookies stolen via XSS, sniffed over HTTP, or leaked in URLs. Attacker replays the cookie = instant login as victim.

🧩 Broken password reset

Predictable reset tokens, sent over email with no expiration, or accessible without verifying the user. Lots of bounty gold here.

Chapter 03 // Real world

📰 When login breaks, internet catches fire

2012

🔗 LinkedIn

117M passwords leaked. Stored as unsalted SHA-1 — crackable in hours. Drove the "everyone just cracks every leaked database" era.

2013

☁️ Yahoo

3 BILLION accounts compromised. MD5 hashes, security questions, backup emails. The largest breach in history until recently.

2022

🔧 Okta breach

Attackers hit Okta's support contractor, used leaked session tokens to impersonate authenticated users at Okta's customers. 2FA didn't save them because session theft bypasses it.

ongoing

🧨 Credential stuffing everywhere

Every major service (Disney+, Spotify, Netflix, Zoom) gets waves of credential stuffing monthly. Source: the 10B+ leaked credentials circulating online.

Chapter 04 // Building auth right

🛡️ Defense checklist

  • Hash passwords with bcrypt, argon2, or scrypt. Never MD5, SHA-1, or plaintext.
  • Rate limit login attempts per IP AND per account. Lockouts + CAPTCHAs for repeat offenders.
  • Enforce MFA — at minimum for admin accounts, ideally for all.
  • Check breached passwords via HaveIBeenPwned API at signup. Block known-bad passwords.
  • Secure session cookies — HttpOnly, Secure, SameSite=Lax.
  • Short-lived tokens — session timeouts, refresh token rotation, expire on password change.
  • HTTPS everywhere — no auth over plain HTTPS alternatives.

🎯 Quick check

A site stores passwords using SHA-256 with no salt. Is this secure?
Yes — SHA-256 is a strong hash.
No — SHA-256 is fast, so GPUs can crack billions of hashes/second. Use bcrypt, argon2, or scrypt instead.
Yes, as long as HTTPS is enabled.
No — you should use MD5 instead.
