IDOR 🎣

Insecure Direct Object Reference.
Change the ID. Get data that isn't yours.

"it's just a number" — how every bounty hunter starts their report. 💸
Chapter 01 // Definition

📖 What is IDOR?

A vulnerability where a web application uses user-supplied input (usually an ID in a URL or request) to look up data — but forgets to verify the user is allowed to access that data.

📝 Think of it like a hotel with predictable room keys

You check into Room 216. Your key card also works on 215, 217, 218 — because the hotel made all the keys with the same pattern and the lock never checks "is this guest actually registered to THIS room?"

A real hotel would refuse. A poorly-coded web app doesn't. It just serves up whoever's data the key (ID) matches.

IDOR = the lock exists, but it doesn't check if the key belongs to you.

The anatomy of an IDOR

# Your legitimate URL — looking at YOUR profile GET /api/users/42/profile # You change 42 → 43 in the URL and hit send GET /api/users/43/profile # Vulnerable response: 200 OK { "name": "Bob Random", "email": "bob@company.com", "phone": "+1-555-0123" } # What SHOULD have happened: 403 Forbidden # you don't own this resource
why it's everywhere
Chapter 02 // Why IDOR prints money

💰 The bug bounty hunter's favorite

🔍 Scanners can't find it

Automated tools don't know what "should" be accessible to whom. They'd need to know your permission model, which they can't. IDOR requires human intuition — which pays.

🌐 It's EVERYWHERE

Every feature showing user-specific data is a candidate: profiles, messages, orders, files, invoices, medical records, DMs, bookings. Thousands of surfaces per app.

💥 Impact is huge

One IDOR on a /users/:id endpoint = mass PII dump. Hit an admin-only action (delete user, grant role) = authorization bypass. Hit a billing endpoint = financial data leak.

🧠 Devs miss it constantly

Auth middleware checks "are you logged in?" — but each endpoint must ALSO check "do you own this specific resource?" Devs forget the second check all the time.

Chapter 04 // Hunting IDORs

🔎 Where IDORs love to hide

The 4 patterns that signal IDOR

  • Sequential integers in URLs/bodies: ?user_id=42, /invoice/1009, {"order":15203}
  • Predictable strings: /users/talha — try /users/admin, /users/root
  • Base64-encoded IDs: eyJ1c2VyIjo0Mn0= → decode, it's {"user":42}, just obfuscation
  • UUIDs that leak in responses: If you see cust-a7f3... for yourself + others in a list, try using theirs in lookup endpoints

The hunting workflow

# 1. Create TWO accounts on the target. Account A (you) and Account B (victim). # 2. Log in as A. Burp captures every request. # 3. Browse everywhere — profile, orders, messages, settings, files. # 4. In Burp's Proxy History, filter requests with an ID parameter. # 5. For each request, swap A's ID for B's ID, resend. # 6. Does the response contain B's data? IDOR. # 7. Does it return 403/401? Good — that endpoint is safe. Move on. # Tools that help: # - Burp AuthMatrix / Autorize plugins (automate IDOR testing) # - Turbo Intruder (fast ID enumeration) # - ffuf (fuzz IDs against endpoint paths)
💡 Burp Autorize plugin is the IDOR hunter's best friend. It replays every request as a second user and flags ones that return different data than they should. Set up once, hunt IDORs on autopilot.

Chapter 06 // The fix

🛡️ How to NOT ship IDORs

Option 1 — Always check ownership

# ❌ BROKEN — trusts whatever ID the user sent @app.route('/api/users/<int:user_id>/profile') def profile(user_id): user = User.query.get(user_id) return jsonify(user.to_dict()) # ✅ FIXED — verifies ownership (or admin) @app.route('/api/users/<int:user_id>/profile') @login_required def profile(user_id): if current_user.id != user_id and not current_user.is_admin: abort(403) user = User.query.get(user_id) return jsonify(user.to_dict())

Option 2 — Don't expose IDs at all

Use the current user's session as the source of truth:

# ✅ Endpoint implicitly uses current_user — no ID needed in URL @app.route('/api/me/profile') @login_required def my_profile(): return jsonify(current_user.to_dict())

Option 3 — Use unguessable IDs

Sequential integers (1, 2, 3...) are easy to walk. Random UUIDs aren't. Note: this is obscurity, not security — always still do the ownership check. But UUIDs raise the bar.

# UUIDs instead of ints /api/users/a7f3c9d2-8b1e-4f62-9c4a-6d8e3b7a2c51/profile
🚨 WAFs won't save you. IDOR is indistinguishable from legitimate traffic — the request is well-formed, the user is authenticated, the URL looks right. It only "looks wrong" if you know the authorization model. This is why WAFs can't detect IDOR; only proper server-side logic can.

🎯 Check yourself

An app uses random UUIDs for user IDs instead of sequential integers. Is it safe from IDOR?
Yes — attackers can't guess UUIDs, so IDOR is impossible.
