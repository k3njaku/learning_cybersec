su 👤

Switch user. Drop into another account's shell
as if you had just logged in as them.

"got their password? now you ARE them." 🎭
Chapter 01 // Definition

📖 What is su?

The su command ("substitute user" — though everyone says "switch user") lets you change your current shell identity to another user. You need their password (unless you're root).

📝 Think of it like wearing someone else's ID badge

You're logged in as talha. You want to poke around as admin. You type su admin, enter their password, and — congrats — every command you run now executes with admin's privileges, access, and file ownership.

Your terminal didn't reboot. No new SSH session. You just swapped identities mid-session.

Basic usage

# Switch to user "bob" talha@box:~$ su bob Password: ******** bob@box:/home/talha$ # NOTE: still in talha's home dir! # Switch AND load bob's full environment talha@box:~$ su -l bob # aka: su - bob, or su --login bob Password: ******** bob@box:~$ # now in /home/bob, bob's PATH, bob's shell, etc.
the -l flag matters
Chapter 02 // The dash matters

🔄 su bob vs su - bob

This tripped me up for months. The difference is subtle but important.

su bob (no dash)

  • Identity changes ✅
  • Working directory stays ❌
  • PATH stays ❌
  • Environment variables mostly stay

Useful when you want to stay where you are but run ONE command as bob.

su -l bob (with dash / --login)

  • Identity changes ✅
  • Lands in bob's home ✅
  • PATH resets to bob's ✅
  • Full login shell simulated ✅

Useful when you want to pretend you SSH'd in as bob — cleaner, more realistic.

💡 Pentester preference: Use su -l (or su -) when you escalate — it gives you the target user's environment variables, aliases, history, and dotfiles. Which tells you a LOT about what they do on this box.
su vs sudo
Chapter 03 // su vs sudo

⚔️ The sibling rivalry

Newcomers confuse these constantly. They do different things.

👤 su

Switch identity for the rest of the session. You become another user; all subsequent commands run as them.

  • Needs: target user's password
  • Long-term identity swap
  • Classic Linux command (since 1970s)

⚡ sudo

Run ONE command as another user (defaults to root), then go back to yourself.

  • Needs: YOUR password, plus permission in /etc/sudoers
  • Short-term elevation, command by command
  • Audit trail: logs every sudo command
🎯 Modern distros: Ubuntu and most desktops disable direct root login AND the root password — forcing you to use sudo. su to root won't work without first setting the root password. sudo -i is the modern replacement ("switch to root interactively").
Chapter 04 // For the attackers

🥷 su in the privesc playbook

  • Post-exploitation pivot: You landed a shell as www-data. You find .bash_history showing admin typed a password. Try su admin with that password — bingo.
  • Leaked creds reuse: Hydra cracked one user's password. Try su into every other user on the box with the same password — password reuse is rampant.
  • Catching the mistake: Developers sometimes add their passwords to scripts or config files. Grep for patterns like password=, PASS=, pwd: — then su into each user.
# Classic post-exploit flow www-data$ cat /var/www/html/config.php | grep -i pass $db_password = "Summer2024!"; www-data$ cat /etc/passwd | cut -d: -f1 root admin bob talha www-data$ su admin Password: Summer2024! admin@box:~$ # 🔥 yep, they reused it admin@box:~$ sudo -l # what can admin do as root?
Chapter 05 // For defenders

🛡️ Making su harder to abuse

  • Disable su for normal users — put admins in the wheel group and configure pam_wheel to restrict su to that group.
  • Long unique passwords — if every user has a 20-char random password, password reuse attacks die.
  • Log su attempts — successful AND failed su events go to /var/log/auth.log. Alert on anomalies.
  • Prefer sudo — command-by-command elevation with audit logs beats long-lived identity swaps.
  • Lock the root account — passwd -l root disables password auth for root entirely. Now nobody can su root.
