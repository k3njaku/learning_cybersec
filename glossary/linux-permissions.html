Chapter 01 // Definition

📖 What are Linux permissions?

Every file and directory on Linux has three types of permissions (read / write / execute) that apply to three categories of users (owner / group / others). That's it. That's the whole system — nine bits total.

📝 Think of every file as a house with 3 keys

🧑‍💼 Owner key — for the person who made it.
👥 Group key — for a specific crew (e.g., the "developers" group).
🌍 Others key — for literally every other user on the system.

Each key unlocks different doors — the read door, the write door, the execute door. The permissions string tells you which keys open which doors.

Reading ls -l output

-rwxr-xr-- 1 talha talha 0 Apr 19 10:37 script.sh ───────── owner group others
  • - at the start = it's a file. (d = directory, l = symlink.)
  • rwx = owner can read, write, execute.
  • r-x = group can read + execute. No write. The - is a "nope" slot.
  • r-- = others can only read.

Chapter 02 // The math (it's easier than it looks)

🧮 Symbolic → numeric

Each permission letter has a number. Add them up per group. Three groups = three digits = chmod 755.

r (read) = 4

Can open the file / list the directory.

w (write) = 2

Can modify the file / add-remove files in directory.

x (execute) = 1

Can run the file / cd into the directory.

💡 Why 4, 2, 1? They're powers of 2 (100, 010, 001 in binary). That means each sum from 0-7 maps to a unique combination of r/w/x. It's a bitmask — one number perfectly encodes which switches are flipped.

Common permission cheat sheet

755 rwxr-xr-x # executables, public scripts (default for /bin) 644 rw-r--r-- # normal files (default for most configs) 700 rwx------ # private directories (e.g., ~/.ssh/) 600 rw------- # private files (e.g., ~/.ssh/id_rsa) 400 r-------- # read-only key material 777 rwxrwxrwx # DANGER. Everyone writes, everyone executes.

Chapter 03 // The pentester angle

🥷 Permissions = privesc

80% of Linux privilege escalation comes down to "someone misconfigured permissions." Here's where to look.

🎯 World-writable scripts

Find a file with -rwxrwxrwx (777) that runs as root (via cron, SUID, or sudo). Edit it → your code runs as root.

find / -perm -002 -type f 2>/dev/null

🗝️ World-readable secrets

Config files, backup archives, password files with "others" read enabled. /etc/shadow readable? Game over — crack the hashes offline.

find / -perm -004 -name '*.conf'

🏴‍☠️ SUID binaries

A binary with the SUID bit (-rwsr-xr-x, the s) runs as its OWNER, not as you. If the owner is root and the binary has a flaw → instant root.

find / -perm -4000 2>/dev/null

📂 Writable PATH directories

If your PATH contains a world-writable directory listed before /usr/bin, drop a binary named ls there. Next time root runs ls, it runs yours.

🚨 chmod 777 is a red flag. When you see it in a tutorial as "the fix" — assume the author didn't understand the actual problem. 777 almost always means the attacker wins eventually.
Chapter 04 // For defenders

🛡️ Principle of least privilege

Grant the minimum permissions needed for the job. Nothing more.

  • Private keys? chmod 600. Anything else and SSH will refuse to use them.
  • Shared scripts? chmod 755. Readable + executable, NOT writable by everyone.
  • User home dirs? chmod 750. Owner full, group read, others locked out.
  • Never leave writeable files in a privileged cron. Ever.
  • Audit with find / -perm -002 -type f every so often.
