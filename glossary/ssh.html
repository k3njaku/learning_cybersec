Chapter 01 // Definition

📖 What is SSH?

SSH (Secure Shell) is a network protocol for securely logging into a remote machine and typing commands as if you were sitting in front of it. The connection is encrypted end-to-end, which is why the "S" matters.

📝 Think of it like a walkie-talkie tunnel

You're sitting at your laptop. There's a server across the internet. SSH digs an encrypted tunnel between you and the server. Everything you type goes through the tunnel; every output comes back through the tunnel. Anyone snooping on the network sees only encrypted gibberish.

Default port: 22. Default client: ssh. Default server: sshd (the "d" is for daemon — the background process that listens for incoming connections).

The anatomy of an SSH command

ssh <user>@<host> # basic ssh -p 2222 talha@10.10.10.5 # custom port ssh -i ~/.ssh/id_rsa root@target # with a specific key
why it matters
Chapter 02 // Why pentesters care

💥 Port 22 is always interesting

🔓 Weak creds = instant shell

Admins deploy a box with root:root or admin:password123. Tools like Hydra and Medusa brute-force SSH at thousands of tries per minute. Sometimes you get lucky.

🗝️ Leaked keys = free access

Developers commit id_rsa private keys to GitHub. Attackers scrape them, use them as SSH keys against the company's servers. Zero authentication needed.

🐛 Old CVEs = full compromise

OpenSSH has had vulns historically (CVE-2024-6387 "regreSSHion", CVE-2023-38408 agent forwarding bugs). Unpatched SSH servers = pre-auth RCE potential.

🔄 Lateral movement

Land a shell on box A, find an SSH key in ~/.ssh/, use it to SSH into box B. This is how attackers hop across an internal network.

Chapter 03 // Real-world carnage

📰 SSH in the headlines

2024

regreSSHion (CVE-2024-6387)

A race condition in OpenSSH < 9.8 allowed unauthenticated RCE as root on millions of Linux servers. Dubbed "one of the biggest SSH bugs ever." Patches rolled out, but unpatched boxes remain.

2024

xz-utils backdoor (CVE-2024-3094)

A malicious maintainer injected a backdoor into xz-utils, which is loaded by sshd via systemd. Would have given an attacker with a specific key full SSH RCE on nearly every Linux server. Caught by a Microsoft engineer noticing a 500ms timing glitch.

ongoing

Mirai botnet

Mirai and its descendants compromise hundreds of thousands of IoT devices by brute-forcing SSH and telnet with a list of ~60 default creds. Still active to this day.

CTF

Every beginner HTB/THM box

Port 22 is open on ~80% of beginner boxes. Grab SSH creds from a web app vuln or scraped service banner, use them to SSH in, escalate from there.

Chapter 04 // For the operators

🛠️ SSH moves you should know

Basic usage

# Connect with password ssh talha@10.10.10.5 # Connect with key (the secure way) ssh -i ~/.ssh/target_key talha@10.10.10.5 # Run a single command remotely ssh talha@target 'whoami; id; hostname' # Copy files with scp (SSH's file cousin) scp myfile.txt talha@target:/tmp/ scp talha@target:/etc/passwd ./passwd_copy

Generating your own key pair

ssh-keygen -t ed25519 -C "talha@laptop" # Creates: # ~/.ssh/id_ed25519 (PRIVATE — never share) # ~/.ssh/id_ed25519.pub (public — paste into authorized_keys on server)
🚨 Never ever share your private key. Not in screenshots. Not in git. Not in Slack. If it leaks, anyone can impersonate you on every server that trusts that key. Regenerate immediately if it's exposed.

The SSH tunnel trick (port forwarding)

Need to reach an internal service (like a database on localhost:5432 of a remote box) from your own machine?

# Forward remote port 5432 → your local port 5432 ssh -L 5432:localhost:5432 talha@target # Now from YOUR machine: psql -h localhost -p 5432 # And you're talking to the remote database through the SSH tunnel.
Chapter 05 // Locking it down

🛡️ How to NOT get pwned via SSH

  • Disable password auth — set PasswordAuthentication no in /etc/ssh/sshd_config. Force key-based auth only.
  • Disable root login — PermitRootLogin no. Make attackers brute-force a normal user first, then sudo.
  • Change the default port — moves you out of the way of drive-by scans. Security theater, but reduces log noise.
  • Use fail2ban — bans IPs after N failed login attempts. Kills brute-force attacks cold.
  • Patch, patch, patch — apt update && apt upgrade openssh-server. regreSSHion-style bugs get patched fast.
  • 2FA — layer TOTP (Google Authenticator) on top of keys via pam_google_authenticator.

🎯 Quick Check

Why do security teams prefer SSH keys over passwords?
Keys are shorter, so they're faster to type.
Keys are mathematically infeasible to brute-force, while passwords can be guessed at scale.
Keys don't need to be stored anywhere.
Keys work over plain HTTP, no encryption needed.
