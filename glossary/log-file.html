Log Files 📝

Every login, every request, every crash — the system writes it all to /var/log/.
Where attackers hide, where defenders look, and where YOU should hunt first.

"Logs don't lie. They also rarely forgive." 🔍
Chapter 01 // Definition

📖 What is a log file?

A log file is a plain-text file where software writes timestamped records of events: who logged in, what URL was hit, what error fired, when a service crashed.

📝 Think of it like a security camera with a notepad

A camera films 24/7 but can't analyze. It just records. Later, when something goes wrong — "who was in this room at 3 AM?" — you rewind. Log files are the same idea, except the tape is infinite, it's text, and you search it with grep instead of a remote.

Where they live

On Linux, almost every log lives in /var/log/. Services each get their own file or directory:

/var/log/auth.log # logins, sudo, SSH attempts /var/log/syslog # general system events /var/log/kern.log # kernel messages /var/log/apache2/access.log # every web hit /var/log/apache2/error.log # web server errors /var/log/nginx/*.log # same but for nginx /var/log/fail2ban.log # blocked brute-force IPs /var/log/ufw.log # firewall drops/allows

Chapter 02 // The log-reading toolkit

⌨️ Commands you'll use daily

tail -f /var/log/...

Follow a log live — new lines appear as they're written. Perfect for watching in real time during an attack.

grep "ERROR" app.log

Search a log for matching lines. -i for case-insensitive, -E for regex, -c to count matches.

less /var/log/syslog

Scroll a big log with pager controls (arrow keys, / to search, q to quit).

journalctl -u sshd

Modern systemd log viewer. Filter by service, time, priority, user. Replaces most /var/log/ browsing on recent distros.

Chapter 03 // Logs for offense

🥷 What pentesters find in logs

🚨 Goldmines hiding in logs:
  • Valid usernames. auth.log shows every SSH login attempt — including successful ones. Now you know which accounts are real.
  • Internal IPs & hostnames. App logs often reveal the private network structure you can't see from outside.
  • Creds in query strings. Lazy devs put tokens/passwords in URLs → those get logged in plaintext access logs. Forever.
  • Vulnerable endpoints. error.log mentions stack traces and paths that shouldn't be public knowledge.
  • Old attack traces. Previous pentesters or bad actors left footprints — scan for UNION, ../, <script>, .php.
Chapter 04 // Logs for defense

🛡️ Spotting an attack from logs

When investigating a breach, these queries are bread-and-butter:

# Failed SSH login attempts by IP (sorted by count) grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn # All 404s from the last hour grep " 404 " /var/log/apache2/access.log | grep "$(date +'%d/%b/%Y:%H')" # Requests with suspicious payloads egrep -i "union|select|<script|\.\./|base64" /var/log/apache2/access.log
🧠 Brain Power
You pop a root shell and want to cover tracks. Why is "just deleting /var/log/auth.log" a bad move?

Three reasons: (1) Most production systems forward logs to a remote SIEM or log server — the local file is just a copy, the originals are already gone to a place you can't reach. (2) A missing or truncated auth.log is itself a huge red flag that triggers alerts. (3) systemd's journald may still have the entries in binary form under /var/log/journal/. Real OPSEC is don't generate the logs in the first place — use living-off-the-land binaries, minimize commands, and avoid logging in as root where possible.
