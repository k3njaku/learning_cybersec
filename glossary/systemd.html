systemd ⚙️

The first process Linux starts at boot. Parent of every other process on the system.
If your service starts automatically — systemd is why.

"PID 1 is the ring-bearer. systemd is the ring." 💍
Chapter 01 // Definition

📖 What is systemd?

systemd is the init system on modern Linux — the first program the kernel launches after boot, which then starts every other service (SSH, web server, databases, logging...). It's PID 1.

📝 Think of it like the conductor of an orchestra

When the concert starts, one person walks on stage first — the conductor. Their job isn't to play music; their job is to start, coordinate, and manage every musician. That's systemd. Every service on your Linux box is a musician; systemd decides when they start, what order, and what to do if one goes silent.

Key job: managing units

Everything systemd manages is called a unit. The most common unit type is a .service — a definition of how to run a background program.

Unit files live in two places:

  • /lib/systemd/system/ — shipped by package installs (don't edit)
  • /etc/systemd/system/ — your overrides and custom services (editable)
the cli
Chapter 02 // systemctl

🎛️ The 5 commands you'll actually use

You never type "systemd" itself — you talk to it via systemctl.

▶️ systemctl start <service>

Start it NOW. Won't survive a reboot unless you also enable.

⏹️ systemctl stop <service>

Stop it NOW. Won't re-disable on boot unless you also disable.

✅ systemctl enable <service>

Make it auto-start every time the system boots. Persists.

❌ systemctl disable <service>

Stop it from auto-starting on boot. Running state unchanged.

ℹ️ systemctl status <service>

Is it running? When did it last start? What's the PID? Recent logs?

🔄 systemctl restart <service>

Stop + start in one command. Use after config changes.

💡 Enable ≠ Start. Enable tells systemd "next time we boot, turn this on." It does NOT start it immediately. Common rookie trap. Use systemctl enable --now <service> to do both in one go.

Real example

# Make Apache survive reboots AND start it now sudo systemctl enable --now apache2 # Check it worked systemctl status apache2 # Nuke it from orbit sudo systemctl disable --now apache2
the security angle
Chapter 03 // Why pentesters care

🥷 systemd as a priv-esc vector

Any service is a unit file. Any unit file is just text. If you can write a unit file that runs as root, you are root at the next boot (or systemctl daemon-reload && systemctl start).

🚨 Classic privesc paths via systemd:
  • World-writable unit file — someone chmod 777'd /etc/systemd/system/cleanup.service. Edit the ExecStart= line to run your shell. Wait for next start → root.
  • Path abuse in ExecStart — ExecStart=backup.sh instead of an absolute path. Drop a malicious backup.sh earlier in PATH.
  • Sudo permission to run systemctl — sudo -l shows you can systemctl start my-thing. Create a user-controlled unit that runs as root.
  • Custom user units — if low-priv user can register units that get started as root, game over.

Enumeration cheat sheet

# List every service systemd knows about systemctl list-units --type=service --all # Find unit files and check their owner/perms ls -la /etc/systemd/system/ /lib/systemd/system/ 2>/dev/null find /etc/systemd/ -writable -type f 2>/dev/null # juicy # View a specific unit file systemctl cat ssh.service # See WHAT started a service (great for chaining) systemctl show ssh.service | grep -E "ExecStart|User"
🧠 Brain Power
Why does ps aux | head always show PID 1 = /sbin/init or /lib/systemd/systemd?

Because the kernel hands off to exactly ONE process after boot — that process becomes PID 1 and is responsible for everything else. Historically this was /sbin/init (SysV init); modern distros symlink that to systemd. PID 1 has two special rules: (1) it can't be killed by SIGKILL (even root can't kill init — the system would panic), and (2) orphaned processes (whose parent died) automatically get re-parented to PID 1.

Chapter 04 // systemd vs cron

⚔️ "Should I use a service or a cron job?"

⚙️ systemd service

  • Long-running background program (web server, DB)
  • Should auto-restart on crash
  • Needs to start at boot and stay running
  • Has dependencies on other services

⏰ cron job

  • Short task run on a schedule (backup, log rotate)
  • Runs, finishes, exits — not a daemon
  • Only needs to fire at specific times
  • Doesn't need dependency tracking
💡 Modern twist: systemd also has .timer units that replicate cron functionality with better logging, timezone handling, and dependency awareness. Many distros are shifting from cron → systemd timers. But cron is still universally installed, so learn both.

🎯 Quick Check

You want nginx to start automatically every time the server reboots. Which command?
