Chapter 01 // Definition

🪪 So what IS a single quote?

In SQL and most programming languages, the single quote ' is a string delimiter. It marks "here starts a chunk of text" and "here ends a chunk of text."

📝 Think of it like quotation marks in English

When you write: She said "hello" to me.

The double quotes tell you: this specific text is what she said. Everything inside = quoted speech. Everything outside = narration.

SQL uses single quotes the same way. They tell the database: "treat everything between these two ticks as a literal string of characters — not code."

Example in plain SQL

SELECT * FROM users WHERE name = 'Talha'

Here, 'Talha' is a string. The apostrophes wrap it. The database reads:

  • ' ← "string starts now"
  • Talha ← the actual text
  • ' ← "string ends now"
the vulnerability
Chapter 02 // Why it breaks everything

💥 The problem: developers glue strings.

When a website wants to search the database for YOU, a naive developer writes code like this:

// PHP (the classic offender) $name = $_GET['name']; $query = "SELECT * FROM users WHERE name = '" . $name . "'";

So if YOU send ?name=Talha, the final query becomes:

SELECT * FROM users WHERE name = 'Talha'

Perfect. Everything works. The app ships. The dev goes home.

But what if... you send a single quote?

Try this: ?name=Tal'ha

SELECT * FROM users WHERE name = 'Tal'ha'

👀 Now the database sees:

  • First ' → string starts
  • Tal → text
  • Second ' → string ENDS (YOUR apostrophe closed it early!)
  • ha → ??? what even is this? not a keyword, not a string... SYNTAX ERROR.
🎯 This is THE tell. If a website errors out when you submit a single quote, you've likely found a SQL injection vulnerability. That error is the website screaming "I took your input and stuffed it into my query like an idiot!"

Chapter 04 // Real-world carnage

📰 Breaches caused by this one character.

The single quote — exploited via SQLi — has been the root cause of some of the most expensive breaches in history.

💳 Heartland Payment Systems (2008)

~130 million credit cards stolen. Albert Gonzalez's crew used SQL injection to get into Heartland's payment processing network. Damages: $140M+.

🎮 Sony Pictures / PSN (2011)

Multiple Sony properties hit with SQL injection by LulzSec. 77 million PlayStation accounts compromised. PSN was offline for 23 days. Estimated cost: $171M.

💼 TalkTalk (2015)

UK ISP hit by SQLi from a 17-year-old. 157,000 customer records stolen. £77M in damages. CEO had to resign.

🏛️ Illinois Board of Elections (2016)

SQLi exposed ~200,000 voter records. Part of broader 2016 election security incidents.

📚 7-Eleven + Nasdaq + Heartland combo (2007-2012)

Same crew, SQLi again. 160 million card numbers across multiple companies. Biggest criminal hacking case in US history at the time.

🎯 Little Bobby Tables (2007, fictional)

xkcd #327. A kid named Robert'); DROP TABLE Students;-- gets registered at school, nukes the student database. The meme that made SQLi famous outside of security circles.

🚨 The pattern: Every one of these boils down to "a user submitted text containing a single quote, and the server pasted it straight into a SQL query." The fix is ALWAYS the same: parameterized queries.

Chapter 05 // For the offensive crowd

🥷 How to USE this knowledge

You're testing an app for vulnerabilities (with permission, obviously). Here's the playbook.

Step 1: Drop a single quote everywhere

Throw ' into:

  • Every URL parameter (?id=1')
  • Every form field (login, search, contact)
  • Every cookie value
  • Every HTTP header the server might read (User-Agent, Referer, X-Forwarded-For)
  • JSON / XML body fields

Step 2: Watch for "tells"

✅ Good signs (SQLi likely)

  • 500 Internal Server Error
  • Stack trace with "SQL syntax" or "ORA-" or "MySQL"
  • The page looks different / broken
  • Different response time

❌ Null signs (not SQLi, probably)

  • Same response regardless of input
  • Obvious client-side validation blocks it
  • Error says "invalid input" but nothing SQL-ish

Step 3: Escalate with payloads

Once the quote breaks something, try these classics (each is its own glossary entry):

// Comment-based bypass '-- // Always-true condition ' OR 1=1-- // Union-based data extraction ' UNION SELECT username, password FROM users-- // Time-based blind ' AND SLEEP(5)--

Chapter 06 // If you're a developer

🛡️ The fix (there is only ONE right answer)

❌ DO NOT try to "sanitize" the single quote by replacing ' with '' or \'. This is called escaping, and it's a whack-a-mole game you WILL lose. Different databases escape differently. Unicode tricks exist. Encoding tricks exist. Just don't.
✅ DO: Use parameterized queries (a.k.a. prepared statements). The database will never confuse user data with SQL code, because they're sent through separate channels.

Quick reference — every major language

// Python (psycopg2 / sqlite3) cursor.execute("SELECT * FROM users WHERE name = ?", (user_input,)) // Node.js (mysql2) conn.query('SELECT * FROM users WHERE name = ?', [userInput]); // PHP (PDO) $stmt = $pdo->prepare('SELECT * FROM users WHERE name = ?'); $stmt->execute([$userInput]); // Java (JDBC) PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE name = ?"); ps.setString(1, userInput); // C# (ADO.NET) cmd.Parameters.AddWithValue("@name", userInput);

🎯 Check Yourself

Why does a parameterized query prevent SQL injection, even if the user types ' OR 1=1--?
Because the database removes all single quotes automatically.
Because the query structure is sent to the database FIRST, and user input is sent separately as pure data — it's never parsed as SQL code.
Because prepared statements are faster.
Because the client-side JS validates the input first.
