📖 glossary · OWASP A01 · the #1 web vuln

Broken Access Control 🔓

The rules about WHO can do WHAT.
When they break, any user does admin things.

"I was logged in, so the server let me." — the attacker, winning. 🎯
Chapter 01 // Definition

📖 What is access control?

Access control is the mechanism that decides: "this user is logged in — but are they ALLOWED to do this specific thing?" Broken access control = that second check is missing or bypassable.

📝 Think of it like a hotel key card system

You swipe your key at your room door → it opens. Swipe it at the hotel safe → it beeps red. Swipe at the penthouse → no entry. Swipe at the server room → definitely no entry.

But imagine a hotel where the key system only checks "is this a real key card?" (yes = open). No check for "which room?", "what floor?", "what time?" Now every guest can walk into every room. That's broken access control.

🧠 Auth vs Authz: Authentication = "WHO are you?" (login). Authorization = "what are you ALLOWED to do?" (access control). Apps nail auth but miss authz constantly.
Chapter 02 // The flavors

🎭 Types of broken access control

🎣 IDOR

Change a number in the URL, access someone else's data. The most common and most lucrative form.

/user/42 → /user/43

📈 Vertical privilege escalation

Regular user → admin. Access an /admin endpoint while logged in as a normal user.

/admin/users.php ← no role check

🤝 Horizontal privilege escalation

One user → another same-level user. Read Bob's messages as Alice.

/messages?to=alice (you're bob)

🚪 Forced browsing

Guessing URLs for pages that should require login.

/admin/, /.git/, /backup.sql

🎭 Role spoofing

Client-side role variables the server trusts. Edit role=user → role=admin in cookies/tokens.

🔓 Missing function-level auth

UI hides buttons for non-admins, but the backend endpoint still works if you call it directly. Classic.

Chapter 03 // Real world

📰 When access control fails

2019

🏦 First American Financial

885 MILLION documents (mortgages, SSNs, bank statements) exposed via IDOR. Sequential URL IDs, no ownership check.

2018

📮 USPS

API leaked info on 60 million postal accounts via missing auth checks. Took USPS a year to fix after being notified.

2021

🏥 Microsoft Power Apps

Default misconfiguration leaked 38 million records across 47 organizations — including COVID contact tracing data and employee SSNs.

2017

🛒 Panera Bread

Customer order lookup had no auth. 37 million customer records (names, emails, partial card numbers) exposed for 8 months.

Chapter 04 // The fix

🛡️ Building proper access control

  • Deny by default. Every endpoint should be locked unless explicitly opened for a specific role/user.
  • Check ownership on EVERY read/write. "Does the session user own this resource?" — if no, 403.
  • Server-side enforcement only. Never trust client-sent roles, flags, or IDs for authz decisions.
  • Use session-based lookups. /api/me/profile is safer than /api/users/:id/profile.
  • Log all authz failures. A spike in 403s = someone's probing for IDORs.
  • Use a framework's RBAC/ABAC. Django permissions, Spring Security, Rails Pundit — don't roll your own.

🎯 Quick check

An app returns a 403 when you change a URL ID to another user's. Is that app vulnerable to IDOR?
Yes — any app with IDs in URLs is vulnerable.
No — a 403 means the server IS checking ownership. That's the correct behavior.
Maybe — try 50 more IDs first.
Yes — 403 is proof the endpoint exists, so it's exploitable.
