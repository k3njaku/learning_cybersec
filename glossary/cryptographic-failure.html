Cryptographic Failures 🔐

When apps try to protect data with crypto, and fail.
Previously called "Sensitive Data Exposure."

"we hashed the password... with MD5." — the phrase that launches thousand-dollar bounties. 💸
Chapter 01 // Definition

📖 What counts as a crypto failure?

OWASP A02. The umbrella for every way sensitive data leaks because of bad crypto choices — or no crypto at all.

📝 Think of it like an envelope for your mail

You write a postcard (cleartext) with your credit card number on it and drop it in the mailbox. Everyone who touches that postcard — postal workers, sorters, the mail carrier — reads your card number.

You could put it in a locked box (encryption). But if the box uses a combination like 1-2-3-4 (weak key), or the box is made of glass (broken algorithm), or you write the combo on a sticky note on the box (hardcoded secret) — you're back to "anyone can read it."

Crypto failures = any time sensitive data isn't properly protected in transit or at rest.

Chapter 02 // The failure modes

💥 Classic ways crypto breaks

🌐 HTTP instead of HTTPS

Login page on plain http:// — anyone on the Wi-Fi reads creds via Wireshark. Shockingly still happens in 2026. Always use HTTPS.

🔓 Weak hashing algorithms

MD5 (broken 2004), SHA-1 (broken 2017) for password storage. Modern GPUs crack MD5 at 200B hashes/sec. Use bcrypt, argon2, or scrypt.

🧂 No salt on password hashes

Same password = same hash. Attackers pre-compute rainbow tables once, crack millions of accounts forever. Salt = unique random per user.

🗝️ Default / weak encryption keys

Hardcoded keys in source (key = "1234"), default secrets in frameworks, or predictable IVs. Any attacker reading the code → full decrypt.

♻️ Using encryption backwards

ECB mode when CBC was needed. Reusing IVs. Confusing encryption with encoding (base64 ≠ encryption). Homegrown crypto is ALWAYS wrong.

💾 Unencrypted backups

Production DB encrypted — backup file on S3 wide open. Attackers love backups; skip the live app entirely.

📝 Sensitive data in logs

Passwords, tokens, credit cards accidentally logged in plain text. Log files get shipped to third-party SaaS → data leak.

🔗 Cleartext tokens in URLs

Password-reset links with tokens in the URL get logged by browsers, referers, CDNs, proxies. Should be one-time-use and short-lived.

Chapter 03 // Real world

📰 Crypto failures that made headlines

2013

🎮 Adobe

153M user passwords leaked. Encrypted with 3DES using ECB mode (not hashed!). Password hints were in cleartext → the whole dataset got cracked by the community.

2012

🔗 LinkedIn

117M SHA-1 (unsalted) password hashes stolen. 90% cracked within weeks. LinkedIn paid for identity protection for years.

2017

🏢 Equifax

Sensitive data (SSNs!) stored unencrypted. 147M Americans' PII leaked. Multi-billion dollar settlement.

2019

☁️ Capital One

106M records leaked from S3 bucket. Data wasn't encrypted at rest — once the bucket was accessible via SSRF, everything was in plaintext.

Chapter 04 // Doing crypto right

🛡️ The crypto hygiene checklist

  • HTTPS EVERYWHERE. HSTS header. Redirect HTTP → HTTPS. Never transmit creds / tokens / session cookies over plain HTTP.
  • Hash passwords with bcrypt, argon2, or scrypt. Never MD5, SHA-1, SHA-256 (too fast), or (heaven forbid) plaintext.
  • Always salt. bcrypt/argon2 salt automatically; you don't have to think about it.
  • Encrypt sensitive data at rest. Databases, backups, logs — full-disk encryption + application-level for PII.
  • Rotate keys periodically. Store them in a secrets manager (AWS KMS, Vault), NOT in source code.
  • Use vetted libraries. Node crypto, Python cryptography, Go crypto/*. Never write your own AES.
  • Use modern TLS (1.2 minimum, 1.3 preferred). Disable old ciphers (RC4, 3DES). Test with SSL Labs.
  • No secrets in logs. Redact passwords, tokens, card numbers before they hit any logger.

🎯 Quick check

A site hashes passwords with SHA-256, no salt. Is this acceptable?
Yes — SHA-256 is a secure algorithm.
No — SHA-256 is too fast (GPUs crack billions/sec) and unsalted hashes enable rainbow tables. Use bcrypt / argon2 / scrypt.
Yes — as long as HTTPS is enabled for login.
No — you should use MD5.
