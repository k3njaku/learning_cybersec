☸️ kubeconfig

The file your computer uses to talk to a Kubernetes cluster.
One leak = full admin control of every app running on it.

"kubectl is the steering wheel. kubeconfig is the key."
Chapter 01 // Definition

📖 What is a kubeconfig?

A YAML file (usually at ~/.kube/config on a developer's laptop) that tells the kubectl CLI how to connect to one or more Kubernetes clusters. It holds three things: cluster URLs, user credentials, and context names.

📝 Your cluster's all-access wristband

Imagine Kubernetes as a giant festival — hundreds of stages (apps), food trucks (databases), lighting rigs (services), all coordinated by a team of organizers (the control plane).

A kubeconfig is a backstage wristband. It proves to the security guard (the API server) that you're allowed in. But unlike a regular wristband, it's not "one-time-use." As long as you have the file, you can walk backstage, rearrange stages, fire performers, or even shut the whole festival down.

If you drop the wristband in the parking lot and someone else picks it up, they're now backstage too.

Chapter 02 // The anatomy

🔬 What a kubeconfig contains

apiVersion: v1 clusters: - cluster: server: https://k8s-prod.acme.internal:6443 certificate-authority-data: LS0tLS1CRUdJTiBDRVJUSUZJQ0FURS0tLS0t... name: prod-cluster users: - name: admin user: token: eyJhbGciOiJSUzI1NiIsImtpZCI6Ik... # OR client cert + key: client-certificate-data: LS0tLS1CRUdJTi... client-key-data: LS0tLS1CRUdJTi... contexts: - context: cluster: prod-cluster user: admin namespace: default name: prod current-context: prod

🌐 server:

The Kubernetes API endpoint. Usually inside a VPC but sometimes public. Leaked file tells attacker EXACTLY where to connect.

🔑 token / client cert

The credential. Tokens rarely expire on dev machines. Certs usually valid for 1 year. Either = full API access.

🛡️ certificate-authority-data

Not sensitive — it's the public CA. But useful to attacker for verifying TLS (no warnings when they connect).

🎭 user permissions

Depends on the cluster's RBAC. Dev laptops often have cluster-admin — game over on leak.

Chapter 03 // The attack playbook

💥 What happens when kubeconfig leaks

# Step 1: verify the connection works kubectl --kubeconfig=stolen-kubeconfig cluster-info # Step 2: see what access you have kubectl --kubeconfig=stolen-kubeconfig auth can-i --list # Step 3: dump every secret in every namespace kubectl --kubeconfig=stolen-kubeconfig get secrets -A -o yaml > all-secrets.yaml # Step 4: read every pod's env vars (more creds) kubectl --kubeconfig=stolen-kubeconfig describe pods -A | grep -A 50 "Environment:" # Step 5: deploy your own malicious pod (mines crypto, pivots to host) kubectl --kubeconfig=stolen-kubeconfig apply -f malicious-pod.yaml # Step 6: escape to the node (mount host filesystem in a privileged pod) # Step 7: lateral movement to other services in the VPC
⚠️ The kicker: By default Kubernetes doesn't alert on kubectl get secrets. Admins only see suspicious API calls if they've explicitly set up audit logging + a SIEM to review it. Most small-to-mid clusters have none.
real world
Chapter 04 // Seen in the wild

📰 kubeconfig leaks that made the news

🤖 Tesla (2018)

An unprotected Kubernetes admin console exposed AWS access keys for production. Attackers used it to mine cryptocurrency on Tesla's servers. Configs were effectively a leaked kubeconfig.

📦 JumpCloud (2023)

Supply-chain attack where attackers gained kubeconfig-equivalent access to a SaaS provider's cluster, then pivoted to customers' environments. Tied to North Korean APT.

🎮 Kubernetes at public-facing game publisher (ongoing research)

Researchers regularly find exposed kubeconfig files in public GitHub repos from smaller studios. Many contain cluster-admin tokens.

🏢 Generic exposure

Shodan + GitHub scans consistently find hundreds of public kubeconfigs at any given time. Most go unreported because researchers avoid the legal risk of testing them.

Chapter 05 // Prevention + recovery

🛡️ How to not leak a kubeconfig

  • .gitignore the whole .kube folder:
    .kube/ kubeconfig *.kubeconfig kube-config
  • Use short-lived credentials — configure the cluster to issue tokens via OIDC (Dex, Keycloak, Okta, Azure AD). Tokens expire in hours, not years. Leaked = attacker has minutes of access.
  • Use kubectl plugins like aws-iam-authenticator, gke-gcloud-auth-plugin — they request fresh creds each use; nothing long-lived in the file.
  • Restrict RBAC — developers should NOT have cluster-admin. Use namespace-scoped roles.
  • Enable audit logging + monitoring — at minimum, alert on get secrets, exec into pods, privileged pod creation, host-path volume mounts.
  • Network-restrict the API server — private endpoint + VPN / bastion / IAM-authenticated tunnel. A leaked kubeconfig is useless if the attacker can't reach server:.
  • If leaked: immediately revoke the user's token/cert in the cluster (kubectl delete csr, rotate service-account tokens, regenerate cluster CA if needed). Review audit logs. Assume full compromise until proven otherwise.

🎯 Quick Check

A developer accidentally commits a kubeconfig to a public GitHub repo. The server: URL points to a PRIVATE API endpoint (accessible only inside the company VPN). What's the attack risk?
No risk — attacker can't reach the API server without VPN access.
Reduced but not zero — the private URL is now known, any future VPN credential leak or SSRF/proxy bug makes the kubeconfig immediately useful. The credential should be revoked regardless.
No risk — private URLs are effectively as good as encryption.
Total risk — Kubernetes private endpoints are always secretly reachable.
