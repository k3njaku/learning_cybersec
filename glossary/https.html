HTTPS 🔒

HTTP wrapped in a TLS tunnel.
The reason your Wi-Fi neighbor can't read your bank login.

"the S is for Secure. finally, an acronym that doesn't lie." ☕
Chapter 01 // Definition

📖 What is HTTPS?

HTTPS = HyperText Transfer Protocol Secure. It's the normal HTTP protocol (what your browser uses to fetch web pages) wrapped inside a TLS (formerly SSL) encrypted tunnel. Runs on TCP port 443 by default.

📝 Think of HTTP vs HTTPS like postcards vs sealed envelopes

Plain http:// is a postcard. Anyone on the route (coffee-shop Wi-Fi router, ISP, a guy with a packet sniffer, the NSA) can read every word — including your login credentials, cookies, and credit card numbers.

https:// is a sealed envelope. Same letter inside, but now encrypted. The route still sees an envelope moving, but can't read what's inside. Only the intended server can decrypt it.

No HTTPS = every password, cookie, and card number in the open. In 2026, HTTP on a login page = automatic OWASP A02 finding.

Chapter 02 // What TLS actually does

🔐 Three guarantees, one handshake

🔒 Confidentiality

Traffic is encrypted. Eavesdroppers see scrambled bytes, not passwords.

✅ Integrity

Messages can't be silently modified mid-flight. MITM proxies that try to inject ads into pages get caught.

🪪 Authenticity

Server proves it's really bank.com via a certificate signed by a trusted Certificate Authority. Fake bank.com → browser screams.

💡 The handshake (simplified):
  1. Browser: "Hi, I want HTTPS. Here are the ciphers I support."
  2. Server: "Cool. Here's my certificate proving I'm bank.com."
  3. Browser checks the cert — signed by a trusted CA? Still valid? Matches hostname? ✅
  4. Both agree on a random session key.
  5. All further traffic encrypted with that key. 🔐
TLS 1.3 does this in one round trip. TLS 1.2 needs two. Old SSL versions are dead — disable them.
Chapter 03 // HTTPS failure modes

💥 When the padlock lies

🌐 Mixed content

Page served over HTTPS but loads JS/images over HTTP. The HTTP resource can be MITM'd → injected code runs in the HTTPS context. Modern browsers block this, but older apps still ship it.

⬇️ HTTPS downgrade attacks

Attacker forces the connection to drop back to HTTP. Defense: HSTS header (Strict-Transport-Security) tells browsers "only ever connect to me over HTTPS."

🕰️ Outdated TLS versions

TLS 1.0 / 1.1 have known weaknesses. SSLv3 is fully broken (POODLE). Still enabled? WAF bypass territory, and compliance violations (PCI-DSS).

🧾 Expired / self-signed certs

Users trained to "click through warnings" = they'll click through a real MITM attack, too. Let's Encrypt gives free certs; no excuse in 2026.

🎭 Missing HSTS

Without HSTS, first visit can be MITM'd before the browser knows to require HTTPS. HSTS preload list solves first-visit too.

🔓 Weak ciphers enabled

RC4, 3DES, EXPORT-grade ciphers. Modern attackers downgrade to these if allowed. Restrict to AEAD ciphers only (AES-GCM, ChaCha20-Poly1305).

Chapter 04 // For testers & defenders

🛠️ Checking a site's HTTPS

Quick tests anyone can run

# The gold standard — SSL Labs free scanner https://www.ssllabs.com/ssltest/analyze.html?d=target.com # Check cert from the CLI openssl s_client -connect target.com:443 -servername target.com < /dev/null # See supported TLS versions + ciphers nmap --script ssl-enum-ciphers -p 443 target.com # Check HSTS and other headers curl -I https://target.com

The defender's must-haves

  • Enable HTTPS everywhere. Let's Encrypt + auto-renewal = free + painless.
  • Redirect HTTP → HTTPS at the load balancer / web server level.
  • Set HSTS (Strict-Transport-Security: max-age=31536000; includeSubDomains; preload).
  • Disable TLS 1.0 / 1.1 and weak ciphers. Use Mozilla's SSL Config Generator.
  • Secure cookies. Add Secure flag so they're only sent over HTTPS.
  • Monitor cert expiry. One expired cert = a full outage. Use automated alerts.
🚨 Common myth: "HTTPS means my site is secure." NO. HTTPS protects data in TRANSIT. It does nothing against SQL injection, XSS, IDOR, or broken auth. You still need all the other defenses. HTTPS is table stakes, not the finish line.

🎯 Quick check

You visit https://bank.com. The browser shows a padlock. Which of these is the MOST valuable guarantee?
The site has no vulnerabilities.
Your traffic is encrypted AND you're talking to the real bank.com (not an impostor).
The bank's servers can't be hacked.
Your password will never be stolen.
