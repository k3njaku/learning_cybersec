apt 📦

The main command for installing, updating, and removing software on Ubuntu, Debian,
Kali, and Parrot. The Linux App Store, but it's a CLI.

"apt install hacktools — and suddenly you have hacktools." 🛠️
Chapter 01 // Definition

📖 What is apt?

apt (Advanced Package Tool) is the command-line package manager on Debian-based Linux distros. It installs, updates, and removes software from trusted repositories.

📝 Think of it like an App Store with signed receipts

You want Firefox. You type apt install firefox. apt looks up "firefox" in its catalog of known software (repositories), downloads it, checks the signature against a trusted publisher key (GPG), and installs it. All dependencies are handled automatically. It's magic — except when it isn't.

The four verbs you'll use daily

🔄 apt update

Refresh the package catalog. Does NOT install or upgrade anything — just tells apt what's available. Run this first, always.

⬆️ apt upgrade

Install the latest versions of everything already installed. After apt update, this actually applies the updates.

📥 apt install <pkg>

Install a new package. Pulls dependencies automatically. You'll type this constantly on a Kali box.

🗑️ apt remove <pkg>

Uninstall. Use apt purge <pkg> to also wipe the config files.

💡 The daily ritual: sudo apt update && sudo apt upgrade -y. Update catalog, upgrade everything, don't ask me again (-y). Every Linux user types this 100 times a year.
how it trusts software
Chapter 02 // Repositories & GPG

🔐 How does apt know what to trust?

apt installs code that will run as root on your system. That's a huge trust decision. apt handles this with two systems: repositories and GPG keys.

Repositories

A repo is just an HTTP/HTTPS server that serves signed package listings and the packages themselves. Your list of trusted repos lives in:

/etc/apt/sources.list # main list /etc/apt/sources.list.d/*.list # drop-in files (better practice)

A typical entry:

deb http://archive.ubuntu.com/ubuntu/ jammy main restricted universe multiverse │ │ │ │ │ └── the URL │ └── components (sub-categories) └── "binary package" └── distribution / codename

GPG keys

Every repo's package list is signed with the repo owner's private key. Your machine holds their public key and uses it to verify: "Yep, this really came from them, and no one modified it."

# Add a third-party repo's signing key (classic, older distros) wget -qO - https://download.sublimetext.com/sublimehq-pub.gpg | sudo apt-key add - # Then register the repo itself echo "deb https://download.sublimetext.com/ apt/stable/" \ | sudo tee /etc/apt/sources.list.d/sublime-text.list # Refresh catalog (now trusted) and install sudo apt update && sudo apt install sublime-text
⚠️ apt-key is deprecated on modern Ubuntu (22.04+). New way: drop the GPG key into /etc/apt/keyrings/ and reference it in the sources file with [signed-by=/etc/apt/keyrings/vendor.gpg]. Fixes a security flaw where any trusted key could sign any package.
the security angle
Chapter 03 // When apt goes wrong

🥷 Supply-chain risk

If an attacker can add their repo to your sources.list — or their key to your trusted keys — they can push software to your machine that runs as root. That's apocalyptic.

🚨 Real attack surface:
  • Untrusted repo additions. Tutorials that say "just run curl http://random-site | sudo bash." Don't. You have no idea what key is being trusted.
  • Repo takeover. If a maintainer's private key leaks, attackers can sign backdoored packages that apt installs silently.
  • Typosquatting. A fake repo with a similar name. Same trick as npm/PyPI supply chain attacks.
  • Abandoned PPAs. Old Ubuntu PPAs (Personal Package Archives) where the original maintainer let the name lapse → anyone can claim it → push updates that your machine auto-installs on next apt upgrade.

Pentest enumeration

On a compromised Linux box, checking what's trusted often reveals interesting things:

cat /etc/apt/sources.list /etc/apt/sources.list.d/* apt-key list # trusted keys (deprecated but still present) ls /etc/apt/keyrings/ # modern keys dpkg -l # everything currently installed
🧠 Brain Power
Why does apt install require sudo, but apt search doesn't?

Installing software modifies system-level directories (/usr/bin/, /etc/, /lib/) — only root can write there. Searching just reads a local catalog in /var/lib/apt/lists/, which is world-readable. Principle of least privilege in action: only the verbs that actually need root ask for it.

Chapter 04 // Useful daily apt commands

🎯 Beyond install/remove

🔎 apt search <term>

Find packages by name or description. Fuzzy-matches.

📋 apt show <pkg>

Full details on a package — version, size, dependencies, description, homepage.

🗂️ apt list --installed

Everything currently installed. Pipe into grep for filtering.

🧹 apt autoremove

Uninstall packages that were installed as dependencies but aren't needed anymore. Run this monthly to reclaim disk.

🎯 Quick Check

You just added a new repository to /etc/apt/sources.list.d/. What MUST you run before you can apt install from it?
apt upgrade
apt update
apt-get refresh
reboot
