UNION Attack 🧱

A SQLi technique where you staple your OWN SELECT onto the dev's query.
Result: you can pull data from any table — including the one with usernames and passwords.

"Two SELECTs enter, one result set leaves."
Chapter 01 // Definition

📖 What is a UNION attack?

In SQL, the UNION keyword combines the results of two SELECT statements into one result set. A UNION attack is when an attacker injects their own SELECT via SQL injection so the DB returns data the app never meant to expose.

📝 The shared cafeteria tray analogy

The DB is a cafeteria. The original query says "give me 3 things: a name, a price, a stock count." The DB grabs a 3-slot tray and fills it.

A UNION attack slides a second order onto the same tray: "also give me 3 more things — a username, a password, an email." The DB obliges. All 6 items (well, 3 sets of 3) come back in one response.

The app — still thinking it's showing products — renders your stolen data as rows on the page. 💀

The canonical UNION injection

original input' UNION SELECT username, password, NULL FROM users--

We close the original string, glue on a UNION SELECT, then comment out whatever trailing SQL the dev had.

the rules
Chapter 02 // Two rules you cannot break

📏 What makes a UNION attack actually work

Rule 1 — Column count must match

Both SELECTs must return the same number of columns. If the original returns 3 columns and your UNION SELECT has 4 → instant SQL error. See the column count lab for how to discover N.

Rule 2 — Column types must be compatible

Each column position must hold a compatible data type across both queries. NULL is compatible with anything — that's why probing with NULLs works so well.

the workflow
Chapter 03 // The full UNION-SQLi kill chain

🔗 From injection point to exfiltrated passwords

  1. Find an injection point. Any user-controlled value that ends up in a WHERE clause.
  2. Count the columns. Use ORDER BY N or UNION SELECT NULL,NULL... until the DB stops erroring.
  3. Find a text-displayable column. Replace each NULL with 'a' one at a time. The slot that renders as "a" on the page is where you'll put your exfiltrated data.
  4. Enumerate tables & columns. Query information_schema.tables and information_schema.columns to find juicy target tables like users.
  5. Exfiltrate. UNION SELECT username || ':' || password, NULL FROM users-- → the page now renders creds.
☕ The big picture: counting columns is step 2. But it's the gateway. Once you can UNION, the rest is just SELECT statements — plain old SQL, weaponized.
🧠 Brain Power
Why can't you just inject UNION SELECT 'hello' and have it work? (Assume the original query has 5 columns.)

Column count mismatch. Your UNION SELECT returns 1 value; the original returns 5. The DB errors instantly. You'd need UNION SELECT 'hello', NULL, NULL, NULL, NULL — five values to match the five slots.

This is why Rule 1 is Rule 1.

defense
Chapter 04 // How to prevent UNION attacks

🛡️ The only real fix

Parameterized queries. Every time. No exceptions.

❌ Broken (concatenation)

// PHP — classic broken $sql = "SELECT name, price FROM products WHERE category = '" . $_GET['category'] . "'";

✅ Fixed (parameterized)

// PHP PDO — the fix $stmt = $pdo->prepare("SELECT name, price FROM products WHERE category = ?"); $stmt->execute([$_GET['category']]);

With parameterized queries, the database knows the value is data, not code. UNION, OR 1=1, comments — none of them escape the value slot. It's a structural fix, not a validation hack.
