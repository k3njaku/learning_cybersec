Recon: Know Your Prey

You don't fire a single line of code until you've looked the target in the eye.

"Amateurs attack. Pros recon."

Skip to code → Why recon? 🧠
Chapter 01.1 // Why bother

🕵️ Why we don't just write code

You could just pip install requests and start hacking away. But you'd waste hours guessing. Recon answers three brutal questions before you write a line:

❓ Can we even reach it?

Is the page public? Does it return 200? Or is it behind a login / paywall / 403 wall?

❓ Is it JS or HTML?

Does the data live in the HTML we download? Or does JavaScript build it in your browser? Huge difference — different tools entirely.

❓ Is anything fighting back?

Cloudflare? WAF? Rate limits? Bot detection? We want to know the enemy before the first shot.

📝 A burglar's analogy

A bad burglar walks up to a house and yanks the door. A good burglar watches the house for a week first. Notices when the owner leaves. Counts the windows. Spots the camera.

Recon is watching the house. Same exact skill as bug bounty recon — different hat, same mind.

the hook
Chapter 01.3 // The hook

🪝 Finding the handle

Every scraper needs a hook — a unique pattern in the HTML that marks the exact thing you want. Miss this step and you'll grab 900 wrong elements.

When I fetched the page, I searched for the company names. Every one is wrapped in exactly this pattern:

<!-- snippet from the page source --> <a href="https://textilepages.com/ambros-etnerprise" title="Ambros Etnerprise" class="company-title-primary company-list-view-title text-strong"> Ambros Etnerprise </a>

That class company-title-primary is the hook. 🎣 Nothing else on the page uses that class. Every single company card has it. That one string is the difference between a working scraper and 4 hours of frustration.

🧠 Brain Power
Why do we use the CSS class as our hook, instead of "find every <a> tag on the page"?

Because the page has hundreds of <a> tags — nav links, footer links, product thumbnails, pagination, "Contact" buttons. If you grabbed all of them, you'd have to filter the junk out yourself.

The class name is the developer's label saying "this is a company title." We steal their label and use it as our filter. Way cleaner, way faster.

👉 Lesson: always look for the most specific, most unique pattern that marks your target data.

recon toolkit
Chapter 01.4 // Your recon toolkit

🧰 How to recon any site yourself

I did this recon for you. Next time, you do it. Three tools, that's it:

1. 👀 The Browser

Right-click → View Page Source. If you see the data in there, it's HTML-rendered (easy). If you don't, it's JS-rendered (harder).

2. 🔬 DevTools

Right-click → Inspect an element. See its HTML tag + class + ID. That's where you find your hook.

3. 🐚 curl

Terminal. curl -I url shows response headers. curl url shows raw HTML — what your script will see.

💻 Try it right now

Run these three commands in your terminal. They're the same ones I used for recon:

# 1. Check if the page is reachable + what server is behind it curl -I https://textilepages.com/textile-companies # 2. Pretend to be a real browser (User-Agent helps bypass basic bot checks) curl -s -A "Mozilla/5.0" https://textilepages.com/textile-companies | head -50 # 3. Search the HTML for our hook — does "company-title-primary" appear? curl -s -A "Mozilla/5.0" https://textilepages.com/textile-companies | grep -c "company-title-primary"
🧠 What grep -c does: counts how many lines contain the pattern. If that number matches "how many companies you see on the page," your hook is solid. If it's 0, the class name might be wrong (or loaded by JS). If it's way higher, the class appears in places other than company titles.
🧠 Brain Power
Run the third curl | grep -c command. You'll likely see a number like 20. Count the companies on the actual page. Does it match?

You should see ~20 (or whatever the "per-page" count is on textilepages). If it matches → your hook is perfect. Every card gets grabbed exactly once.

This is how pros verify a selector before writing any parsing code. Two seconds, saves two hours.

Chapter 01.5 // Check yourself

🎯 Quick check

🎯 Question 1

You fetched a target page with curl and looked at the HTML. The product prices you want to scrape are nowhere in the HTML source. What's most likely going on?
The website is broken.
The prices are rendered by JavaScript after the page loads — you'll need a browser automation tool (Selenium/Playwright) or find the underlying API.
You need to pay for the data.
Your internet is slow.

🎯 Question 2

What is the "hook" in scraping recon?
The URL of the page.
The password for the site.
A unique, specific HTML pattern (usually a class or ID) that marks the exact data you want — used later to filter parsed HTML.
A Python library.

🎯 Question 3

A site returns HTTP 200 OK but its Server header says cloudflare. Can you still scrape it with basic tools?
No, never.
Often yes — Cloudflare is only blocking if you get a 403, a "Checking your browser…" page, or a CAPTCHA. A 200 + real HTML means you're through. A normal User-Agent usually seals the deal.
You need to hack Cloudflare first.
Only with paid proxies.

You know the target, you found the hook, you know what tools to use.
