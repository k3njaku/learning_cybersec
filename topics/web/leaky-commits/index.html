Leaky Commits 🗑️

You're hunting API keys. Good. But attackers hunt everything else too.
Private keys. Database URLs. Terraform state. Internal hostnames. Schemas. Screenshots.

"An API key is a door. A leaked terraform.tfstate is a floor plan of the entire building."

Chapter 01 // The Mental Shift

🎯 The burglar-and-recycling-bin analogy

Picture a burglar casing a house. You think they want the safe, right? The cash? The jewelry?

📝 What a smart burglar ACTUALLY wants

They want the recycling bin.

Why? Because inside it they find:

📬 Envelopes with the homeowner's name (social engineering target)
🏦 Bank statements showing balance + branch
📦 Amazon boxes showing what tech is inside
💊 Medicine packaging revealing a health condition (blackmail angle)
✈️ Boarding pass stubs (they know when you'll be away)

None of it is "valuable" on its own. All of it together is a heist plan.

GitHub leaks work the same way. Your scanner is looking for the "cash" (API keys). But skilled attackers also want the envelopes — the config files, schemas, hostnames, screenshots, commit messages. Any one of those is a shortcut.

🧠 Brain Power
You're an attacker. You can steal ONE of these two things. Which is more dangerous — a password to the admin panel, OR a complete database schema (table names, column names, relationships)?

Trick question — the schema is usually more dangerous over time.

A stolen password gets rotated within 24 hours once someone notices. One shot, one kill.

A leaked schema lets you craft surgical SQL injection payloads against EVERY vulnerable input on the site, for YEARS. Even if they patch one SQLi, you know exactly what to dump next time. The schema itself is rarely considered "sensitive" and almost never rotated.

👉 Attackers play the long game. Leaks that describe the system are often worth more than leaks that unlock the system.

Chapter 02 // The Leak Tiers

🗂️ What actually gets leaked

Think of leaks as a menu — 7 categories, ranked by how often scanners MISS them.

🔑 Tier 1 — API Keys & Tokens

sk_live_, AIza, AKIA... You already hunt these. Scanners catch 90%+.

🗝️ Tier 2 — Non-key Credentials

SSH private keys, DB connection strings, kubeconfigs, cloud JSON service accounts. Catch rate: ~40%.

🗺️ Tier 3 — Infrastructure Blueprints

terraform.tfstate, .env files, Ansible inventories, CI/CD configs. Catch rate: ~10%. Attackers LOVE this tier.

💾 Tier 4 — Data Leaks

Customer CSVs, test fixtures with real emails, screenshots of admin panels, database dumps. Catch rate: almost 0%.

🧬 Tier 5 — Intellectual Property

Source code, DB schemas, algorithms, API docs, Postman collections. Not "credentials" but gold for targeted attacks.

😅 Tier 6 — Context Leaks

Commit messages spilling secrets, .env.example with real values, developer shell histories. Hilarious. Catastrophic.

🎲 Tier 7 — Weird Stuff

JWTs in tests, Android keystores, browser extension signing keys, VPN configs, OAuth client secrets. Rare but devastating.

☕ The takeaway

Tiers 1-2 are what your scanners find. Tiers 3-6 are where real pentest alpha lives.

Chapter 04 // Receipts

🔥 This actually happens. A lot.

Real breaches. Real companies. Real numbers. All traced back to "non-key" leaks on GitHub.

🚗 Uber (2016)

Engineer pushed code to a private GitHub repo with AWS credentials hardcoded. Attacker found it, pulled data for 57 million users + 600,000 drivers.

Uber paid the attackers $100K to stay quiet. FTC later fined them $148 million. The CSO was personally indicted.

🚙 Toyota (2022)

Subcontractor pushed code to a public GitHub repo containing an access key to a customer data server. The repo sat public for FIVE YEARS.

296,019 T-Connect customer records exposed — emails + customer management numbers.

⭐ Mercedes-Benz (2024)

Employee accidentally published a GitHub token on GitHub. Researchers (RedHunt Labs) used it to access "unrestricted" internal source code, including blueprints, design docs, SSO + SSH keys, database connection strings, and cloud credentials.

The token wasn't even the actual secret. It just unlocked the store of secrets.

🏛️ US Department of Defense (2023)

Contractor's public Azure storage bucket was indexed for weeks containing 3 TB of internal military emails, including ones with security clearance info.

Not technically GitHub, but same root cause: "we didn't think anyone would look."

☕ NetworkChuck Moment: Every one of these started with "oops" and ended with front-page news. Your responsible disclosure work — the thing you're already doing at 10K/month — literally prevents these. You're not just catching keys. You're catching the next headline.

Chapter 05 // Hunt Like a Pro

🎯 GitHub queries that find gold

Save these. Chain them with a target domain when you have one. Just run them raw when you're doing mass scanning.

💼 The "credentials not-a-key" set

// SSH private keys — instant shell on someone's server "BEGIN RSA PRIVATE KEY" "BEGIN OPENSSH PRIVATE KEY" filename:id_rsa // Database connection strings with passwords baked in "jdbc:mysql://" password "mongodb+srv://" password "postgres://" password // Config files with secrets filename:.env filename:.env.production filename:wp-config.php filename:database.yml password filename:web.config connectionString

🗺️ The "blueprint" set (Tier 3 gold)

// Terraform state = full infrastructure map filename:terraform.tfstate filename:*.tfvars // Kubernetes cluster control filename:kubeconfig "kubectl config" token // AWS credentials files "aws_secret_access_key" filename:credentials filename:.aws/credentials // CI/CD configs revealing pipelines filename:.gitlab-ci.yml password filename:Jenkinsfile credentials

💾 The "raw data" set (Tier 4)

// SQL dumps — yes, the entire database extension:sql INSERT INTO users extension:sql "CREATE TABLE" // Customer data filename:users.csv email filename:customers.xlsx // Editor swap files devs forget about filename:.env.swp extension:bak password

🎯 The "target-first" combo (BEST for pentest)

This is the killer pattern. Always pair the target with a leak type:

// Example: client is tjwholesale.pk "tjwholesale.pk" filename:.env "tjwholesale.pk" shpat_ # Shopify admin token "tjwholesale.myshopify.com" api_key "tjameel@tjwholesale.pk" # find their commits
💡 Pro tip: Document the queries that returned ZERO results too. When you hand a client a pentest report, "I checked these 20 things and found nothing" is more credible than "I didn't find anything." Evidence of absence ≠ absence of evidence.
defense
Chapter 06 // The Fix

🛡️ How devs stop this

❌ Broken — the classic oops

# .env (committed to the repo by accident) DATABASE_URL=postgres://admin:hunter2@prod-db.company.com:5432/users STRIPE_SECRET=sk_live_51abc... AWS_ACCESS_KEY_ID=AKIA... # .gitignore is empty, or this file isn't in it

✅ Fixed — the one-line habit

# .gitignore — commit this file instead .env .env.* !.env.example *.pem *.key id_rsa* terraform.tfstate* .aws/ kubeconfig *.sqlite *.db *.sql.backup .bash_history

✅ Better — pre-commit hooks

Install gitleaks or trufflehog as a pre-commit hook. They block commits that contain secret-shaped content before they land on GitHub:

# install pre-commit + gitleaks pip install pre-commit brew install gitleaks # or: go install github.com/gitleaks/gitleaks/v8@latest # .pre-commit-config.yaml in the repo root repos: - repo: https://github.com/gitleaks/gitleaks rev: v8.18.0 hooks: - id: gitleaks # activate pre-commit install

🚨 Already leaked? Here's the checklist

Deleting the file from the repo is NOT enough. Git history is forever. The attacker already cloned it. You must:
  1. Rotate the leaked credential immediately (new key, new password, revoke old)
  2. Rewrite git history with git filter-repo or BFG Repo-Cleaner
  3. Force push the rewritten history
  4. Contact GitHub Support to purge cached views of the old commit
  5. Review logs for any suspicious activity using the old credential
  6. Notify users if the leak exposed their data (PECA 2016 / GDPR / CCPA depending on jurisdiction)

🎯 Quick Check

A developer accidentally committed a file called terraform.tfstate to a public GitHub repo. A scanner that ONLY looks for API key patterns will probably miss it. Why is it still dangerous?
It isn't — tfstate files are just infrastructure configs, no secrets.
It's a full map of their infrastructure — IPs, resource names, sometimes plaintext secrets in outputs — letting an attacker target the real systems directly.
It's only dangerous if the developer also leaked an SSH key in the same commit.
Terraform encrypts tfstate by default, so it's safe.
