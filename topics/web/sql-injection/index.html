📝 Plain English, please

A web app is basically a middleman between you and a database.

When you search "funny cat shirts," your browser sends that text to the server, and the server builds a SQL query like:

SELECT * FROM products WHERE name = 'funny cat shirts'

SQL injection = you sneak SQL code INTO that search box, and the server runs it as if it were part of the original query. 🤯

It's like whispering extra instructions to the bouncer so he lets you into the VIP room.

🧠 Brain Power
Why does this even work? Why doesn't the database just "know" which part is user input and which part is code?

Because by default, the web app just glues strings together. It takes your input and stuffs it into a SQL sentence. The database then parses that whole sentence as one instruction — it has no idea which parts came from you and which came from the developer.

👉 This is the root cause. And the fix (parameterized queries) is basically telling the database: "Hey, THIS part is data, not code." We'll get to that at the end.

why should you care
Chapter 02 // Why attackers LOVE this

💥 The impact is... chef's kiss.

SQLi isn't just "oops you saw a hidden product." It's been responsible for some of the biggest breaches in history.

🔓 Stolen Data

Passwords, credit cards, SSNs, private messages. Anything in the database is fair game.

🗑️ Destroyed Data

DELETE, DROP, UPDATE. An attacker can modify or wipe data permanently.

🔑 Full Auth Bypass

Log in as anyone — including admin — without knowing the password.

🖥️ Server Takeover

In some configs, SQLi leads to arbitrary code execution on the actual server. Game over.

☕ NetworkChuck Moment: Companies pay bug bounty hunters thousands of dollars for finding SQLi in production. It's one of the OG vulnerabilities. OWASP Top 10. Always.

Chapter 03 // The Detective Work

🔍 How do you even FIND SQLi?

You poke at every input and watch for weird reactions. Like prodding a lion to see if it's real or stuffed.

🕵️ The SQLi Smell Test

Throw each of these at every input field, URL parameter, cookie, header:

  • ' (just a single quote) — if the app errors out = ding ding ding
  • OR 1=1 — does the page return way more data than expected?
  • OR 1=2 — does the page return nothing?
  • time delays — does the page hang for 5 seconds?
  • OAST payloads — does your server get a ping you didn't expect?

If any of these change the behavior, you've probably got SQLi. 🎯

Where does it hide?

Most people think SQLi only happens in WHERE clauses. WRONG. It can live in:

  • UPDATE statements (in the value OR the WHERE)
  • INSERT statements (in the inserted values)
  • Table names or column names
  • ORDER BY clauses (these are sneaky — prepared statements don't always help here!)

Different input formats, same attack

Think SQLi only happens in URL parameters like ?id=1? Nope. It can happen in:

POST /search HTTP/1.1 Content-Type: application/json { "productId": "' OR 1=1--" }

Or even XML, using encoding to slip past WAFs:

<stockCheck> <productId>123</productId> <storeId>999 &#x53;ELECT * FROM users</storeId> </stockCheck>

☝️ That &#x53; is the letter "S" encoded — useful for slipping past filters that block the word "SELECT".

💡 Why does -- work?

In SQL, two dashes is a SQL comment — meaning "ignore the rest of the line."

So when you inject Gifts'--, the bouncer reads:

WHERE category = 'Gifts'--' AND released = 1

Everything after the dashes is deleted from the bouncer's brain. The "only show released products" rule? Gone. Now you see unreleased products too.

🧠 Brain Power
Why did we need the single quote ' before the --? What happens if you forget it?

The original query is WHERE category = 'Gifts' AND released = 1. The apostrophes are a string delimiter — they tell SQL "this text is one string."

If we just inject Gifts--, the query becomes WHERE category = 'Gifts--' AND released = 1. The -- is inside the string, so it's just treated as literal text. No comment, no bypass. 💀

We inject the apostrophe to close the string early, THEN write our comment. It's the key move.

⚠️ Warning: Be careful with OR 1=1. If that same input is also used in an UPDATE or DELETE query elsewhere, you might accidentally update/delete EVERYTHING. On real targets, always start with -- style bypasses.

🧠 Brain Power
The PortSwigger lab specifically asks you to log in as administrator. Why wouldn't ' OR 1=1-- in the username field always work for that?

' OR 1=1-- returns all users, and the app typically logs you in as the first one returned, which is usually whoever was registered first — not necessarily the admin.

That's why administrator'-- is the cleaner attack: you explicitly target that username and comment out the password check. 🎯

Chapter 06 // Stealing From Other Tables

🏴‍☠️ UNION Attacks

What if the data you want isn't in the products table? What if it's in the users table? Enter: UNION.

🔗 UNION in 30 seconds

UNION is a SQL keyword that says: "run a second query and glue the results onto the first one."

So if the app runs:

SELECT name, description FROM products WHERE category = 'YOU'

You can inject:

' UNION SELECT username, password FROM users--

And now the page displays... usernames and passwords, side by side with product names. 🎣

⚙️ Two rules for UNION:
1️⃣ Both queries must return the same number of columns
2️⃣ The data types must be compatible (usually: find a column that's a string type)

We'll go deep on UNION in its own dedicated page (determining column counts with ORDER BY n or UNION SELECT NULL,NULL,..., finding string columns, chaining data into one column with concatenation, etc.). This is a whole art form. 🎨

Chapter 07 // Hacking in the Dark

🕶️ Blind SQL Injection

Sometimes the app doesn't show you query results OR errors. It's silent. But you can still hack it — by asking yes/no questions.

🎲 The 3 Blind Techniques

  • Boolean-based: "If the admin's password starts with 'a', show the welcome message. Otherwise, show nothing." You infer the answer from the page's behavior.
  • Time-based: "If true, sleep 5 seconds." Watch your stopwatch. ⏱️
  • Out-of-Band (OAST): "Hey database, DNS lookup this domain I own with the password as the subdomain." Your server receives a ping with the answer. 🛰️
🎧 Think of it like this: You're talking to someone through a wall. You can't hear them, but you can tell if they knocked once (yes) or twice (no). Blind SQLi is literally asking the database yes/no questions one character at a time. It's slow. It's tedious. It's beautiful.

Tools like sqlmap automate this. We'll build out a full blind SQLi simulator on its own page when we get to that PortSwigger section. 💪

the antidote
Chapter 08 // The Good Guy Chapter

🛡️ How to Prevent SQLi

One fix. One answer. Memorize this forever: parameterized queries.

❌ Broken (string concatenation)

String query = "SELECT * FROM products WHERE category = '" + input + "'"; // input = "Gifts' OR 1=1--" → RIP database Statement statement = connection.createStatement(); ResultSet rs = statement.executeQuery(query);

✅ Fixed (parameterized)

PreparedStatement statement = connection.prepareStatement( "SELECT * FROM products WHERE category = ?" ); statement.setString(1, input); // input is now DATA, never treated as SQL code. Safe ✅ ResultSet rs = statement.executeQuery();

🧠 Why it works

With a prepared statement, the SQL structure is fixed and sent to the database FIRST. Then the user input is sent separately, as pure data.

The database already knows "this is a SELECT with one string parameter." Your input can never change that — it can only fill in the blank. The apostrophe is just... an apostrophe. 🤷

⚠️ Gotcha: Prepared statements can't parameterize table names, column names, or ORDER BY. For those, you MUST use an allowlist (e.g., if (sortField not in ['name','price','date']) reject;).
