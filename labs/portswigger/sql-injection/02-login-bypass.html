Chapter 01 // Refresher

🧠 Login forms, but make them vulnerable

A typical login query looks like this:

SELECT * FROM users WHERE username = 'alice' AND password = 'hunter2'

If that query returns one or more rows, the app thinks "cool, credentials match" and logs you in. If it returns zero rows → rejected.

📝 The bouncer analogy (remix)

You're at a club. The bouncer (the SQL query) has a clipboard and needs to tick TWO boxes: your name AND your password.

Normal person says: Name: alice, Password: hunter2 → bouncer checks both → maybe lets in.

Attacker says: Name: administrator'--, Password: literally anything

Bouncer reads: "Name is administrator, uhh there's a comment, guess I'll ignore the password check 🤷" → lets in.

🧠 Brain Power
We used ' -- in Lab 1 to comment out AND released = 1. Why would the same trick work on a login form?

Because the login query has the exact same shape: a WHERE with multiple AND-joined conditions. If we can close the username string and comment out the rest, the AND password = '...' check gets skipped completely.

SQL injection is almost always just recognizing the shape of a query and applying the same 3–4 tricks.

Chapter 03 // Tools

🔧 Browser is fine. curl is better.

🌐 Browser login form

Works perfectly — just type the payload into the username field. But the POST is "hidden" in a form submit, which makes understanding it less obvious.

💻 curl (recommended for this one)

You see the HTTP POST. You see the Set-Cookie header that confirms you're logged in. Teaches you how auth actually works.

🚫 No Burp. Yes, even for POSTs. curl does POST requests too (that's what -d is for).
the solve
Chapter 04 // The Guided Solve

🛠️ Step-by-step

Open the lab & find the login form browser

Click "Access the lab" → click "My account" (top right). You'll see a username + password form.

Why scout first: you need to see the form to know which parameters to attack. Always reconnaissance before injection.

Peek at how the form submits devtools

Open Firefox DevTools (F12) → Network tab. Submit the form with junk creds (test / test). In the network panel, click the login request:

  • Method: POST
  • URL: /login
  • Body: csrf=...&username=test&password=test

Note the csrf token — it's unique per session and the server will reject requests without it.

Why check first: knowing the exact parameter names (username, password, csrf) means our curl request will match what the server expects.

Guess the backend query theory

Almost certainly:

SELECT * FROM users WHERE username = '[username]' AND password = '[password]'

If that returns a row, you're logged in as whoever the row says. So we want to make the query return the administrator row while skipping the password check.

Why theorize: with this mental model, the payload is obvious. Close the username string, inject a comment, nuke the password check.

Craft the payload theory

Submit this username:

administrator'--

(A space at the end helps with some DBs — PortSwigger's lab DB accepts both.)

Final query becomes:

SELECT * FROM users WHERE username = 'administrator'-- ' AND password = 'whatever'

Password field: doesn't matter. We're commenting it out. Literally put "x" in there.

Why it works: the DB only executes up to the --. It finds the administrator row, sees it's a match, returns it. The app logs you in as admin because that's who the row belongs to.

Fire it in the browser browser

Back in the browser login form:

  • Username: administrator'--
  • Password: x

Click log in. You'll be redirected to the administrator's "My Account" page. Banner turns green. 🎉

Why browser here: the CSRF token is auto-attached by the form. Simplest path. curl requires us to grab + forward that token manually.

(Optional) reproduce in curl shell

For the reproducible version, you need to (a) grab the CSRF token + session cookie from the login page, (b) POST the injection with them attached:

bash · ~/labs
$ LAB="https://YOUR_LAB" # 1) Grab login page, save cookies, extract csrf CSRF=$(curl -sc cookies.txt "$LAB/login" | grep -oP 'name="csrf" value="\K[^"]+') echo "csrf=$CSRF" # 2) POST the injection, reusing cookies curl -sb cookies.txt -c cookies.txt -i "$LAB/login" \ --data-urlencode "csrf=$CSRF" \ --data-urlencode "username=administrator'-- " \ --data-urlencode "password=x" | head -20 HTTP/2 302 location: /my-account?id=administrator set-cookie: session=LoggedInAsAdmin...; Secure; HttpOnly; SameSite=None ...

The 302 redirect to /my-account?id=administrator is the win. You just logged in as admin from the command line.

Why this matters: when you start doing real bug bounty, you'll want to automate things. POC-as-a-curl-command is the gold standard for bounty reports.
dissection
Chapter 05 // Payload Dissected

🔬 Why "administrator'-- " works

SELECT * FROM users WHERE username = 'administrator'-- ' AND password = 'x'

administrator

The username we want to become. This is the actual row we're after in the users table.

'

Closes the opening quote early. Now username = 'administrator' is a complete, valid condition.

--

Comment. Everything after this is invisible to the DB — including the entire password check.

💡 Same trick, new form. Notice this is the identical payload from Lab 1, applied to a different input. That's the pattern: recognize the query shape, apply the shape-breaking payload.

Chapter 06 // Real-world bounties

💰 Auth bypass in the wild

🐦 Twitter (2010)

A famous early SQLi in Twitter's auth flow — researchers showed they could log in as other users by manipulating query parameters. Led to multiple platform fixes.

🏦 Banking / fintech startups

A disturbing number of bug bounty reports on HackerOne show SQLi-based auth bypass in fintech/lending platforms. Payload patterns are essentially what you just did.

⚠️ Ethics reminder: only do this on your own apps, PortSwigger labs, or targets with an explicit bug bounty scope. SQLi on production without authorization = federal crime in most countries.
