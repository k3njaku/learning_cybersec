Chapter 01 // Refresher

🧠 SQL injection — the 2-minute recap

A website is a middleman. You type, the server builds a sentence in SQL (the database's language), and the database answers. SQL injection = sneaking your own words into that sentence.

📝 The restaurant analogy

Imagine a waiter (the web app) going to the kitchen (the database) with your order written on a notepad.

You say: "I'll have the chicken."

Waiter writes down: SERVE chicken TO customer WHERE table = 5

Now watch: what if your "order" was chicken'; DROP TABLE tables; --?

The waiter (not very smart) just copies it: SERVE chicken'; DROP TABLE tables; -- TO customer WHERE table = 5

The kitchen reads that whole thing as TWO commands. Serves the chicken. Then drops the tables. 💀

🔑 Three facts you need for THIS lab

1. Single quote = string boundary

In SQL, ' marks where text starts and ends. Break that, you break the query. See single quote.

2. -- = comment

Two dashes + a space tell SQL "ignore everything after this." Your delete key for the rest of the query. See SQL comment.

3. OR 1=1 = always true

1 always equals 1. When you OR this into a WHERE clause, it matches every row.

🧠 Brain Power
If a query is SELECT * FROM products WHERE category = 'Gifts' AND released = 1 and you want EVERY product returned (released or not), what single logical thing do you need to change about this query?

You need to cancel the AND released = 1 condition. Two ways to do it:

① OR-in something that's always true → makes the whole WHERE clause true for every row.
② Comment out the AND released = 1 part → the database never evaluates it.

This lab uses option ②. We'll close the string with ', then use -- to nuke the rest of the line.

Chapter 04 // The Guided Solve

🛠️ Step-by-step

Every step: DO THIS → SEE THIS → HERE'S WHY. If something in "See this" doesn't match, re-check "Do this."

Click "Access the lab" on PortSwigger browser

You'll get a unique URL that looks like this:

🔒 https://0a1b2c3d4e5f.web-security-academy.net/

That bold prefix is YOUR instance's ID — it's different every time. Call this YOUR_LAB for the rest of this walkthrough.

Why: every PortSwigger lab spins up a fresh, isolated container. Your instance can't be attacked by others and yours can't attack theirs.

Click any category filter on the home page browser

Click "Gifts" (or any other category). Watch your URL change to something like:

🔒 https://YOUR_LAB/filter?category=Gifts

There it is — the injection point. That category=Gifts parameter is almost certainly being dropped straight into a SQL query.

Why: any time user input appears in the URL and changes the database output, you should smell SQL. This is the smell test every bug hunter runs.

Guess the server-side query theory

Based on what we see (a filter that picks one category AND hides unreleased items), the backend query is almost certainly:

SELECT * FROM products WHERE category = 'Gifts' AND released = 1

Our plan: make that AND released = 1 disappear. If we can close the string after Gifts and then comment out the rest of the line, the database will never check released — and we'll get every product.

Why: you don't attack blind. You build a mental model of the query first, THEN craft a payload that specifically breaks that model. This is the #1 skill that separates "tried payloads from a list" from "actual SQL injection."

Craft the payload theory

We want the final query to become:

SELECT * FROM products WHERE category = 'Gifts'--' AND released = 1

Breakdown of what we inject after Gifts:

  • ' — closes the opening quote started by the developer
  • -- (two dashes + a space) — tells SQL "ignore the rest of this line"

Everything after -- (the AND released = 1) gets treated as a comment. The database just returns all products in the Gifts category — released or not.

Why the space after --? Most DB engines (Oracle, Postgres, MSSQL) require a space after the dashes for it to count as a comment. MySQL requires a space OR tab OR newline. When in doubt, add the space — safest across dialects.

URL-encode the payload (only the special characters) theory

A URL can't contain a raw space or certain characters — they need to be percent-encoded. For our payload:

Char Encoded Why
' %27 Most browsers accept raw ' in the URL, but %27 is safer in curl / scripts.
(space) %20 or + URL can't contain literal spaces.

So our payload becomes: Gifts'--+ (using + for space) or Gifts%27--%20 (fully encoded). Both work.

Why encode at all? Sometimes the browser helpfully encodes for you. Sometimes it doesn't. Encoding manually removes ambiguity and makes your payload identical whether you use a browser or curl.

Fire the attack (browser) browser

Paste this into your address bar (replace YOUR_LAB with your instance):

🔒 https://YOUR_LAB/filter?category=Gifts'--

Hit Enter. The page reloads and you should see more products than before — including ones marked unreleased. At the top of the page, the "Lab is solved" banner turns green. 🎉

Why browser first: you see the rendered product grid immediately. No parsing HTML, no grep — just eyeballs and dopamine.

Verify with curl (optional but sweet) shell

In a terminal:

bash · ~/labs
$ curl -sG "https://YOUR_LAB/filter" \ --data-urlencode "category=Gifts'-- " | grep -oE '<h3>[^<]+' | head <h3>Gift Card <h3>Gift Wrap <h3>Discount Codes (unreleased) <h3>Mystery Gift Box (unreleased) ...

The -sG flags mean "silent + convert --data to GET". --data-urlencode handles the encoding for us. The grep extracts product names from the HTML.

Why curl matters: reproducible. If you ever need to prove the bug to someone (bounty report, screenshot, video), you can paste the exact one-liner. Browsers hide way too much.
dissection
Chapter 05 // Payload Dissected

🔬 The winning string, one character at a time

SELECT * FROM products WHERE category = 'Gifts'-- ' AND released = 1

Gifts

Legitimate-looking data. Makes the query succeed so the app doesn't error out. Any real category name works.

'

The killer character. Closes the developer's opening quote early, prematurely ending their string.

--

Two dashes + space. Says "ignore the rest of this line." The AND released = 1 part never gets evaluated.

💡 Pattern to memorize: value' -- is the single most common SQLi payload on Earth. It's in the top 10 of every wordlist, every scanner, every pentest report.

Chapter 06 // Real-world bounties

💰 "But has this actually happened?"

Yes. Often. With big payouts.

🏦 Starbucks — $4,000

Researcher Eugene Lim found an SQLi in a Starbucks internal endpoint. Same pattern: a filter parameter got dropped into a WHERE clause. Payload was effectively ' OR '1'='1'-- .

🎮 Various — HackerOne reports

Search hacktivity for "sql injection" — you'll see dozens of reports, many paying $1k–$10k, built on this exact pattern.

🏛️ 2023 MOVEit breach

A single SQLi in Progress Software's MOVEit file-transfer product led to one of the biggest supply-chain breaches of 2023. ~2,600+ organisations hit, 85M+ individuals' data leaked. Starts the same way: user-controlled input reaching a query.

🛒 Equifax / Heartland / Sony Pictures

Multiple landmark historical breaches started with SQLi. The technique is 20+ years old and STILL top 10 of OWASP.

☕ Mindset shift: every "where does the user's text end up in a query?" is potential money. Bug bounty is literally this, but on production apps that haven't been pentested.
